# 앱 센터 기획서

## 1. 문서 목적

이 문서는 `My_Web_OS`의 `Package Center`, `Sandbox`, `Widget Store`, 그리고 향후 `앱 제작/배포/실행 생태계`를 하나로 묶는 최상위 기획 문서다.

이 문서의 목표는 다음과 같다.
- 현재 구현된 `샌드박스 앱 + 패키지 센터`를 어떤 방향으로 키울지 정의한다.
- `앱`, `위젯`, `샌드박스`, `런타임`, `패키지`의 관계를 명확히 정리한다.
- 사용자가 앱을 만들고, 수정하고, 실행하고, 배포하는 흐름을 제품 레벨에서 설계한다.
- 개발자가 어디서부터 어떤 순서로 구현해야 하는지 단계별 로드맵을 제공한다.

---

## 2. 한 줄 비전

`My_Web_OS`의 앱 센터는 단순한 앱 목록 창이 아니라,

**앱을 설치하고, 만들고, 수정하고, 실행하고, 위젯으로 배치하고, 나중에는 배포까지 할 수 있는 로컬 앱 플랫폼**

이 되는 것을 목표로 한다.

쉽게 말하면:
- 지금은 `앱 서랍`
- 앞으로는 `앱 제작소 + 앱 런처 + 앱 관리소 + 위젯 허브`

가 되어야 한다.

---

## 3. 현재 상태 요약

현재 프로젝트에는 이미 다음 요소가 있다.

### 3.1 Package Center
- 설치된 샌드박스 앱 목록을 보여준다.
- 앱 메타데이터, 버전, 권한, 엔트리 정보를 표시한다.
- 선택한 앱을 데스크톱 창에서 실행할 수 있다.

### 3.2 Sandbox Runtime
- `server/storage/inventory/apps/<app-id>/` 구조로 앱을 저장한다.
- `manifest.json` 기반으로 앱을 읽는다.
- `/api/sandbox/:appId/...`를 통해 앱 자산과 앱 데이터 접근을 제어한다.
- 앱별 데이터 공간과 권한 기반 접근 제어가 있다.

### 3.3 hello-sandbox
- 샌드박스 앱 구조를 보여주는 예제 앱이다.
- 개발자가 샌드박스 앱을 어떤 형태로 구성해야 하는지 참고할 수 있다.

### 3.4 Widget 구조
- 위젯은 별도의 라이브러리와 상태 저장 체계를 가진다.
- 현재는 샌드박스 앱과 완전히 통합되어 있지 않다.

즉, 현재는 `실행 가능한 기초 골격`은 있고, 앞으로는 이를 `플랫폼`으로 확장해야 하는 단계다.

---

## 4. 왜 지금 이 방향이 중요한가

현재 구조는 이미 상당히 좋은 출발점이다.

- 로컬 파일 시스템과 시스템 정보에 접근할 수 있다.
- 데스크톱 UI, 창 시스템, 상태 저장 구조가 있다.
- 샌드박스 앱을 앱 목록에 합치는 흐름이 이미 존재한다.

이 말은 곧,

**앱 생태계를 얹을 수 있는 바닥공사 자체는 이미 끝났다는 뜻**이다.

이제 필요한 것은:
- 앱을 더 쉽게 만들 수 있게 하고
- 앱을 더 안전하게 실행하게 하고
- 앱과 위젯을 하나의 개념으로 정리하고
- 사용자도 “내가 이 환경 안에서 기능을 추가할 수 있다”는 감각을 갖게 만드는 것이다.

## 5. 핵심 제품 철학

앱 센터는 다음 철학 위에서 설계한다.

### 5.1 로컬 우선
- 이 플랫폼은 클라우드 앱스토어보다 `내 PC/NAS/홈랩 안에서 확장되는 로컬 플랫폼`에 가깝다.
- 사용자는 인터넷이 없어도 자신의 앱을 만들고 실행할 수 있어야 한다.

### 5.2 샌드박스 우선
- 앱 추가가 곧 시스템 전체 신뢰를 의미해서는 안 된다.
- 기본적으로 앱은 제한된 권한 안에서 돌아가야 한다.
- 시스템 접근은 항상 `명시적 권한`과 `제어된 API`를 통해서만 가능해야 한다.

### 5.3 제작 가능한 플랫폼
- 패키지 센터는 설치된 앱을 “보는 곳”에서 끝나면 안 된다.
- 사용자가 직접 앱을 만들고, 수정하고, 복제하고, 실험할 수 있어야 한다.

### 5.4 앱과 위젯의 통합
- 앱과 위젯은 본질적으로 같은 패키지의 다른 표현이어야 한다.
- 차이는 “창으로 열리느냐”, “바탕화면에 붙느냐”이지, 제작 방식은 최대한 통일한다.

### 5.5 HTML 우선, Python은 단계적
- 가장 먼저 잘 지원해야 할 것은 `HTML/CSS/JS` 기반 패키지다.
- Python은 가능하지만, 즉시 서버 실행형으로 열면 보안/운영 부담이 크므로 단계적으로 확장해야 한다.

## 6. 최종 지향 구조

장기적으로는 아래처럼 정리되는 것이 가장 이상적이다.

### 6.1 사용자에게 보이는 개념
- `앱 센터`: 모든 패키지를 관리하는 곳
- `패키지`: 설치 가능한 단위
- `앱`: 창으로 열리는 패키지
- `위젯`: 바탕화면에 붙는 패키지
- `하이브리드`: 앱과 위젯 둘 다 가능한 패키지

### 6.2 내부 개념
- `패키지 저장소`: 패키지 파일과 메타데이터 저장
- `샌드박스 런타임`: 패키지를 격리 실행
- `권한 게이트웨이`: 파일, 데이터, 네트워크, 시스템 API 접근 통제
- `패키지 편집기`: manifest, 코드, 자산 수정
- `미리보기/실행기`: 즉시 테스트

### 6.3 한 문장으로 정리
- `Package Center`는 제품 이름
- `Sandbox`는 실행 방식
- `App/Widget`은 패키지의 표현 형태

## 7. 개념 재정의

혼동을 줄이기 위해 내부 개념을 다음처럼 재정의한다.

### 7.1 Package
배포/설치/수정/실행 가능한 최소 단위다.

예:
- 메모 앱
- 시스템 모니터 위젯
- 날씨 위젯
- 미디어 리모컨 앱

### 7.2 Sandbox
패키지가 실행되는 제한된 환경이다.

의미:
- 앱은 독립된 파일 공간을 가진다.
- 허용된 권한만 사용할 수 있다.
- 직접 시스템 전체를 건드리는 것이 아니라, 게이트웨이를 통해 기능을 요청한다.

### 7.3 App
창으로 열리는 패키지다.

### 7.4 Widget
바탕화면이나 대시보드 영역에 붙는 패키지다.

### 7.5 Hybrid Package
하나의 패키지가 앱으로도, 위젯으로도 동작할 수 있는 형태다.

예:
- 시계: 위젯 가능, 앱 가능
- 리소스 모니터: 위젯 가능, 앱 가능
- 노트 앱: 기본은 앱, 간단 요약 위젯 지원 가능

## 8. 제품 방향의 핵심 결론

이 문서의 가장 중요한 결론은 다음 세 가지다.

### 8.1 Package Center는 “앱 제작소”까지 포함해야 한다
지금은 목록과 실행 중심이지만, 앞으로는 아래 기능이 들어가야 한다.

- 새 패키지 생성
- 기존 패키지 수정
- 패키지 복제
- manifest 편집
- 권한 설정
- 코드 편집
- 즉시 미리보기
- 내보내기/가져오기

### 8.2 HTML/CSS/JS를 1순위 런타임으로 삼아야 한다
이유:
- 가장 구현 난이도가 낮다.
- 가장 직관적이다.
- iframe/정적 자산 구조와 잘 맞는다.
- 위젯과 앱 모두 재사용이 가능하다.

### 8.3 Widget는 Sandbox Package와 통합하는 것이 맞다
위젯은 별도 시스템이 아니라,

**패키지의 표시 모드 중 하나**

로 정리하는 것이 가장 좋다.

## 10. 권장 manifest 진화 방향

현재 manifest는 단순한 구조다. 앞으로는 조금 더 풍부해져야 한다.

권장 예시:

```json
{
  "id": "hello-sandbox",
  "title": "Hello Sandbox",
  "description": "Example package",
  "version": "1.0.0",
  "type": "hybrid",
  "runtime": "sandbox-html",
  "icon": "LayoutGrid",
  "author": "local",
  "entry": {
    "app": "index.html",
    "widget": "widget.html"
  },
  "permissions": [
    "app.data.list",
    "app.data.read",
    "app.data.write"
  ],
  "window": {
    "width": 960,
    "height": 720,
    "minWidth": 480,
    "minHeight": 320
  },
  "widget": {
    "defaultWidth": 240,
    "defaultHeight": 180,
    "resizable": true
  }
}
```

### 핵심 필드 설명
- `type`: `app`, `widget`, `hybrid`
- `runtime`: `sandbox-html`, `sandbox-js`, `sandbox-pyodide` 등
- `entry.app`: 창 모드 엔트리
- `entry.widget`: 위젯 모드 엔트리
- `permissions`: 허용 요청 목록
- `window`: 앱 창 정보
- `widget`: 위젯 기본 크기/설정

## 11. 런타임 전략

### 11.1 1단계: HTML/CSS/JS Sandbox
가장 먼저 완성해야 할 런타임이다.

지원 방식:
- `index.html`
- 정적 자산
- iframe 렌더링
- 부모와 제한된 브리지 통신

장점:
- 구현이 쉽다
- 위젯과 앱 둘 다 가능하다
- 미리보기 구현이 자연스럽다

### 11.2 2단계: JS API Bridge 강화
단순 정적 페이지가 아니라, 샌드박스 앱이 제한된 API를 호출하게 만든다.

예:
- `window.webos.data.read()`
- `window.webos.data.write()`
- `window.webos.openApp()`
- `window.webos.notify()`

즉, 앱은 직접 시스템을 만지는 것이 아니라 `제공된 공식 브리지`만 사용한다.

### 11.3 3단계: Python 지원
Python은 두 가지 방식이 있다.

#### 브라우저형 Python
- 예: `Pyodide`
- 장점:
  - 비교적 안전
  - 샌드박스와 잘 맞음
  - 교육용/실험용에 좋음
- 단점:
  - 무거움
  - 로컬 시스템 제어는 제한적

#### 서버 실행형 Python
- 예: 별도 프로세스/worker 실행
- 장점:
  - 진짜 Python 앱처럼 강력함
- 단점:
  - 보안/격리/리소스 관리 부담 큼

### 11.4 추천 결론
- 1차: `HTML/CSS/JS`
- 2차: `JS Bridge`
- 3차: `Pyodide`
- 4차: 필요시 서버 실행형 Python

즉, Python은 가능하지만 `첫 단계에서 바로 여는 기능이 아니라, 플랫폼이 안정된 뒤 붙이는 기능`으로 가는 것이 맞다.

## 12. Widget와 Sandbox의 통합 전략

제 의견으로는 `통합하는 것이 맞다`.

### 이유

위젯과 샌드박스 앱은 본질적으로 같은 패키지다.

차이:
- 앱: 창으로 열린다
- 위젯: 바탕화면에 붙는다

공통점:
- manifest가 있다
- 코드가 있다
- 권한이 있다
- 상태 저장이 필요하다
- 사용자와 상호작용한다

### 권장 구조

`Widget`를 별도 특수 시스템으로 계속 유지하기보다, 아래로 흡수한다.

- 모든 위젯은 패키지다
- 일부 패키지는 위젯 슬롯을 가진다
- 데스크톱은 `widget-capable package`를 로드한다

### 결과
- 제작 방식 통일
- 편집기 재사용
- manifest 통일
- 권한 체계 통일
- 패키지 센터 하나로 전체 관리 가능

## 13. 강한 샌드박스를 도커 없이 구현할 수 있는가

완전히 도커급은 아니어도, 충분히 강한 실용적 격리는 구현 가능하다.

### 가능한 방향
- 앱별 데이터 루트 분리
- iframe sandbox 속성 강화
- CSP 적용
- 공식 브리지 API만 허용
- 권한 선언 없는 기능은 차단
- 네트워크 접근 allowlist
- 파일 접근은 앱 데이터 루트 우선
- 중요 시스템 API는 사용자 승인 필요
- 감사 로그 기록

### 중요한 원칙
- 앱은 직접 시스템 API를 호출하지 않는다
- 앱은 “기능 요청”만 하고, 실제 실행은 게이트웨이가 판단한다

즉, 구조는 아래처럼 가야 한다.

`패키지 코드 -> 샌드박스 브리지 -> 권한 검사 -> 서버 게이트웨이 -> 실제 작업`

이 구조를 지키면 도커가 없어도 상당히 강한 제어력을 가질 수 있다.

## 14. 이상적인 보안 모델

### 14.1 권한 카테고리

초기 권장 권한:
- `app.data.list`
- `app.data.read`
- `app.data.write`
- `app.ui.notify`
- `app.ui.open`
- `app.widget.pin`

중기 권장 권한:
- `fs.read.scoped`
- `fs.write.scoped`
- `network.fetch.allowlist`
- `system.metrics.read`
- `cloud.read`

고위험 권한:
- `terminal.exec`
- `docker.control`
- `settings.write`
- `system.process.read`

고위험 권한은 기본적으로 샌드박스 앱에 바로 주지 않는 방향이 좋다.

### 14.2 권한 UX
- 설치 시 권한 요약 표시
- 실행 시 민감 권한 승인
- 패키지 센터에서 권한 재검토 가능

## 15. 제안하는 API 확장 방향

현재 `/api/packages`는 읽기 중심이다. 앞으로는 패키지 CRUD와 편집 흐름까지 확장해야 한다.

권장 API:

### 패키지 목록/생성
- `GET /api/packages`
- `POST /api/packages`

### 패키지 상세
- `GET /api/packages/:id`
- `PUT /api/packages/:id`
- `DELETE /api/packages/:id`

### manifest 편집
- `GET /api/packages/:id/manifest`
- `PUT /api/packages/:id/manifest`

### 파일 트리/내용 편집
- `GET /api/packages/:id/files`

## 19. 무엇을 하지 말아야 하는가

초기 단계에서 아래는 피하는 것이 좋다.

### 바로 서버 실행형 Python을 열기
- 너무 강력하다
- 보안 위험이 크다
- 관리 비용이 높다

### Widget와 Package를 별개 제품으로 계속 키우기
- 중복 구조가 많아진다
- 유지보수가 어려워진다

### manifest 중심 UX만 제공하기
- 초보자가 접근하기 어렵다
- 마법사 + 고급 편집 모드 구조가 더 낫다

### 샌드박스 앱에 강한 시스템 권한을 바로 주기
- 플랫폼 신뢰를 깨뜨린다
- 관리자 기능은 별도의 고신뢰 앱/권한 모델로 가는 편이 좋다

## 20. 최종 권장 결론

이 프로젝트의 앱 센터는 다음 방향으로 가는 것이 가장 좋다.

### 제품 결론
- `Package Center`를 `앱 제작/관리 허브`로 키운다
- `Sandbox`는 패키지 실행 표준으로 유지한다
- `Widget`는 패키지의 한 표현 형태로 통합한다
- `HTML/CSS/JS`를 1순위 런타임으로 삼는다
- `Python`은 2차 확장 기능으로 붙인다

### 개발 결론
- 먼저 패키지 CRUD와 편집기를 만든다
- 다음으로 미리보기와 템플릿 시스템을 만든다
- 그 뒤 위젯을 패키지로 통합한다
- 마지막에 브리지 API, 권한 UX, Python을 확장한다

### 한 문장 최종 정리

**My_Web_OS의 앱 센터는 “설치된 앱 목록”이 아니라, 로컬에서 앱과 위젯을 만들고 실행하는 샌드박스 기반 플랫폼으로 발전해야 한다.**

## 21. 바로 다음 액션 제안

가장 먼저 시작할 구현 작업은 아래 네 가지다.
