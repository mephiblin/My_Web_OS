# 수정해야 하는 영역 정리

## 1) Package Center Preflight 체크 항목 중복 표시

- **영역 이름**: Package Center > Install/Update Preflight 리뷰 렌더링
- **이유**:
  - `dependencyChecks`와 `compatibilityChecks`가 동일한 원본(`dependencyCompatibility.checks`)을 우선 참조하고 있어,
    UI에서 두 배열을 합쳐 렌더링할 때 같은 항목이 중복 표시될 수 있음.
  - 사용자 입장에서 사전검토 결과 신뢰도가 떨어질 수 있음.
- **파일 경로**:
  - `client/src/apps/package-center/PackageCenter.svelte`
- **스크립트 라인(기준)**:
  - `dependencyChecks` 정규화: **399~405**
  - `compatibilityChecks` 정규화: **410~415**
  - 합쳐서 렌더링하는 구간: **1287** (`[...installReview.dependencyChecks, ...installReview.compatibilityChecks]`)
- **권장 수정 방향**:
  - `dependencyChecks`와 `compatibilityChecks` 데이터 소스를 분리하거나,
  - 단일 리스트로 통합해 중복 없이 렌더링.

## 2) Preflight 실패 시 설치 진행 가능 상태

- **영역 이름**: Package Center > Preflight 실패 시 실행 제어
- **이유**:
  - preflight API 호출 실패 시 fallback 리뷰를 생성하는 흐름에서 `blocked`가 기본적으로 차단 상태가 아닐 수 있음.
  - 그 결과 `Approve & Install` 버튼을 통해 설치가 진행될 수 있어, “사전검토 강제” 정책과 충돌 가능.
- **파일 경로**:
  - `client/src/apps/package-center/PackageCenter.svelte`
- **스크립트 라인(기준)**:
  - preflight 실패 fallback 처리: **514~520**
  - 실행 차단 조건: **543~545** (`installReview.blocked && !installReview.forcePolicyBypass`)
- **권장 수정 방향**:
  - preflight API 실패 시 `blocked = true`로 강제하거나,
  - “검토 불가 상태” 별도 상태값을 도입해 실행 버튼 비활성화.

## 참고 계약(연관 백엔드)

- **영역 이름**: Registry Preflight API
- **파일 경로**:
  - `server/routes/packages.js`
- **스크립트 라인(기준)**:
  - preflight 엔드포인트: **1907~2056** (`POST /registry/preflight`)
  - 실행 가능 판단 필드: **2039~2042** (`executionReadiness.ready`, `executionReadiness.blockers`)
- **참고 이유**:
  - 프론트 차단 조건을 백엔드의 `executionReadiness`와 일치시키면 정책 일관성이 올라감.
