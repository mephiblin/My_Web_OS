아래는 대화 내용을 바탕으로 **웹OS 설계 문서**처럼 정리한 첨삭본이야. 바로 기획서나 개발 문서에 넣을 수 있도록 문장과 구조를 다듬었어.

# 웹OS 구성 체계: 코어 시스템, 코어앱, 애드온

## 1. 기본 개념

웹OS는 기능의 중요도와 역할에 따라 크게 세 가지 계층으로 구성한다.

```text
코어 시스템
  OS의 생존, 보안, 계정, 설정, 파일 접근, 복구를 담당하는 핵심 영역

코어앱
  사용자가 OS 설치 직후 기본적인 파일과 콘텐츠를 열고 다룰 수 있게 하는 기본 앱 영역

애드온
  OS의 기본 동작에는 필수적이지 않지만, 사용자의 목적과 취향에 따라 기능을 확장하는 선택형 영역
```

이 구조의 핵심은 **OS의 안정성과 확장성을 분리하는 것**이다.

코어 시스템은 작고 견고해야 하며, 코어앱은 기본 조작에 필요한 최소 기능을 제공해야 한다. 애드온은 사용자의 필요에 따라 설치, 제거, 교체할 수 있는 확장 기능으로 설계한다.

---

# 2. 전체 설계 철학

웹OS의 기능 분류는 단순히 “중요한가, 중요하지 않은가”로 나누어서는 안 된다. 어떤 사용자에게는 마크다운 편집기, Git 클라이언트, 캘린더, 음악 플레이어가 매우 중요할 수 있다. 그러나 그것들이 OS가 부팅되고, 로그인하고, 파일을 탐색하고, 시스템을 복구하는 데 반드시 필요한 것은 아니다.

따라서 분류 기준은 다음 질문에서 출발해야 한다.

> **이 기능이 없으면 OS 자체를 사용할 수 없는가, 아니면 사용자의 작업 경험을 확장하는 기능인가?**

이 기준에 따라 웹OS는 다음 철학을 가진다.

> **코어 시스템은 OS의 생존을 책임지고, 코어앱은 기본 사용성을 책임지며, 애드온은 사용자의 작업 세계와 개성을 확장한다.**

즉, 코어는 작고 안정적으로 유지하고, 확장은 애드온을 통해 유연하게 제공한다.

---

# 3. 코어 시스템

## 3.1 정의

**코어 시스템**은 웹OS가 정상적으로 작동하기 위해 반드시 필요한 기반 기능이다.

사용자가 OS를 부팅하고, 로그인하고, 설정을 변경하고, 파일에 접근하고, 문제가 발생했을 때 복구할 수 있도록 하는 핵심 구성 요소가 여기에 포함된다.

코어 시스템은 앱처럼 보일 수 있지만, 실제로는 OS의 기반 기능이다. 예를 들어 설정 화면이나 파일 탐색기는 UI를 가진 앱처럼 보이지만, 시스템을 제어하고 유지하는 핵심 기능이므로 코어 시스템에 속한다.

---

## 3.2 범위

코어 시스템에는 다음과 같은 기능이 포함된다.

```text
- 설정
- 계정 관리
- 권한 관리
- 세션 관리
- 파일 시스템 접근 계층
- 파일 탐색기
- WebDAV
- 터미널
- 네트워크 설정
- 알림 시스템
- 앱 설치 및 삭제 관리자
- 업데이트 시스템
- 작업 관리자
- 로그 및 진단 도구
- 복구 모드
- 보안 정책 관리자
```

---

## 3.3 기준

다음 조건에 해당하면 코어 시스템으로 분류한다.

```text
- 없으면 OS를 정상적으로 사용할 수 없다.
- 없으면 로그인, 설정, 파일 접근, 복구가 어렵다.
- 시스템 전체의 보안이나 권한과 직접 연결된다.
- 장애 발생 시 복구를 위해 반드시 필요하다.
- 다른 앱이나 애드온이 의존하는 기반 API를 제공한다.
- 사용자가 삭제하거나 교체하면 OS 안정성이 크게 흔들릴 수 있다.
```

---

## 3.4 철학

코어 시스템은 **작고, 안정적이며, 신뢰 가능해야 한다.**

코어 시스템이 지나치게 커지면 OS 전체가 무거워지고, 업데이트 위험이 커지며, 보안 취약점이 늘어난다. 따라서 코어 시스템은 반드시 필요한 기능만 포함해야 한다.

코어 시스템의 설계 원칙은 다음과 같다.

```text
- 최소 구성
- 높은 안정성
- 강한 보안
- 복구 가능성
- 애드온으로부터의 독립성
- 실패 시에도 OS 전체가 무너지지 않는 구조
```

코어 시스템은 웹OS의 뼈대이며, 애드온이나 코어앱이 문제가 생기더라도 계속 살아 있어야 한다.

---

# 4. 코어앱

## 4.1 정의

**코어앱**은 사용자가 웹OS를 설치한 직후, 기본적인 파일과 콘텐츠를 열고 다룰 수 있게 해주는 기본 애플리케이션이다.

코어앱은 OS의 생존 자체에 필수적인 것은 아니지만, 사용자가 기본적인 작업을 수행하기 위해 필요하다.

예를 들어 이미지 파일을 열거나, 동영상을 재생하거나, 텍스트 파일을 수정하는 기능은 OS의 핵심 시스템 기능은 아니지만, 기본 사용성을 위해 제공되어야 한다.

---

## 4.2 범위

코어앱에는 다음과 같은 앱이 포함될 수 있다.

```text
- 이미지 뷰어
- 영상 플레이어
- 텍스트 편집기
- 기본 오디오 플레이어
- 기본 PDF 뷰어
- 기본 압축 해제 도구
```

다만 코어앱은 고급 기능까지 모두 포함할 필요는 없다. 오히려 기능을 최소화하고 안정성을 높이는 것이 좋다.

예를 들어 텍스트 편집기는 코어앱이 될 수 있지만, 코드 에디터는 애드온으로 분리하는 것이 적절하다.

```text
텍스트 편집기
  - .txt, .log, .conf 파일 열기
  - 간단한 수정과 저장
  - 기본적인 찾기/바꾸기

코드 에디터
  - 문법 강조
  - Git 연동
  - LSP 지원
  - 터미널 통합
  - 확장 플러그인
```

이 경우 텍스트 편집기는 코어앱, 코드 에디터는 애드온으로 분류한다.

---

## 4.3 기준

다음 조건에 해당하면 코어앱으로 분류한다.

```text
- OS 설치 직후 기본 파일을 열 수 있어야 한다.
- 사용자가 흔히 접하는 파일 형식을 처리한다.
- 기본 보기, 재생, 편집 기능을 제공한다.
- 고급 기능보다는 안정성과 단순성이 중요하다.
- 삭제되더라도 OS는 작동하지만, 기본 사용성이 크게 떨어진다.
```

---

## 4.4 철학

코어앱은 **기본 사용성을 보장하는 최소한의 앱**이어야 한다.

사용자가 파일 탐색기에서 흔한 파일을 열었을 때 “아무것도 할 수 없다”는 느낌을 받지 않게 하는 것이 코어앱의 목적이다.

그러나 코어앱이 너무 많은 기능을 포함하면 코어 시스템처럼 무거워지고 관리가 어려워진다. 따라서 코어앱은 기본 기능에 집중하고, 고급 기능은 애드온으로 분리하는 것이 좋다.

코어앱의 철학은 다음과 같다.

```text
- 기본 파일을 열 수 있게 한다.
- 기본 조작을 가능하게 한다.
- 기능보다 안정성을 우선한다.
- 고급 기능은 애드온으로 확장한다.
- 사용자가 다른 앱으로 대체할 수 있는 여지를 남긴다.
```

---

# 5. 애드온

## 5.1 정의

**애드온**은 웹OS의 기본 동작에는 필수적이지 않지만, 사용자의 목적, 취향, 직업, 작업 방식에 따라 기능을 확장하는 선택형 구성 요소이다.

애드온은 없어도 OS가 부팅되고, 로그인되고, 파일을 탐색하고, 기본 작업을 수행하는 데 큰 문제가 없어야 한다. 하지만 설치하면 사용자의 생산성, 편의성, 개인화, 전문 작업 능력을 크게 향상시킬 수 있다.

애드온은 단순히 “중요하지 않은 앱”이 아니다. 어떤 사용자에게는 애드온이 매우 중요한 작업 도구가 될 수 있다. 다만 OS의 생존에 필수적인 기능은 아니라는 점에서 코어와 분리된다.

---

## 5.2 애드온의 핵심 철학

애드온의 철학은 다음 문장으로 요약할 수 있다.

> **애드온은 코어를 대체하지 않고, 코어 위에서 사용자의 작업 세계를 확장한다.**

애드온은 선택 가능해야 하며, 안전하게 설치·비활성화·삭제될 수 있어야 한다. 또한 시스템의 안정성과 사용자의 데이터 소유권을 침해해서는 안 된다.

애드온의 핵심 원칙은 다음과 같다.

```text
- 선택 가능해야 한다.
- 삭제 가능해야 한다.
- 코어 시스템을 망가뜨릴 수 없어야 한다.
- 필요한 권한만 요청해야 한다.
- 설치 실패나 실행 오류가 OS 전체에 영향을 주면 안 된다.
- 사용자 데이터와 앱 데이터를 명확히 분리해야 한다.
- 다른 애드온이나 코어앱으로 대체 가능해야 한다.
```

---

# 6. 애드온의 범위

애드온은 크게 네 가지 유형으로 나눌 수 있다.

```text
1. 일반 애플리케이션형 애드온
2. 파일 핸들러형 애드온
3. 시스템 확장형 애드온
4. 위젯 및 패널형 애드온
```

---

## 6.1 일반 애플리케이션형 애드온

독립적으로 실행되는 일반 앱이다. 사용자의 작업 목적에 따라 선택적으로 설치한다.

예시는 다음과 같다.

```text
- 마크다운 편집기
- 노트 앱
- 캘린더
- 할 일 관리
- 음악 플레이어
- 이미지 편집기
- PDF 주석 도구
- 다이어그램 편집기
- 스프레드시트 뷰어
- 프레젠테이션 뷰어
- 데이터베이스 클라이언트
- API 테스트 클라이언트
- RSS 리더
- 메일 클라이언트
```

이들은 없어도 OS는 정상적으로 작동하지만, 사용자의 작업 환경을 풍부하게 만든다.

---

## 6.2 파일 핸들러형 애드온

특정 파일 형식이나 MIME 타입을 열고, 편집하고, 미리보기할 수 있게 하는 애드온이다.

예시는 다음과 같다.

```text
- .md, .markdown → 마크다운 편집기
- .pdf → PDF 뷰어 또는 PDF 주석 도구
- .csv → CSV 뷰어
- .svg → SVG 뷰어 또는 편집기
- .json → JSON 뷰어
- .yaml, .yml → YAML 편집기
- .zip, .7z, .tar.gz → 압축 관리자
- .epub → 전자책 리더
- .sqlite → SQLite 브라우저
```

파일 핸들러형 애드온은 파일 탐색기와 자연스럽게 연결되어야 한다.

예를 들어 사용자가 `.md` 파일을 더블클릭하면 다음과 같은 흐름이 가능하다.

```text
1. 파일 탐색기가 확장자 또는 MIME 타입을 확인한다.
2. 연결된 기본 앱을 찾는다.
3. 설치된 마크다운 편집기가 있으면 실행한다.
4. 없으면 관련 애드온 설치를 제안한다.
```

---

## 6.3 시스템 확장형 애드온

시스템 UI나 동작에 영향을 주는 확장 기능이다. 일반 앱보다 권한이 강할 수 있으므로 신중하게 관리해야 한다.

예시는 다음과 같다.

```text
- 클립보드 관리자
- 스크린샷 도구
- 화면 녹화 도구
- 런처 확장
- 검색 인덱서 확장
- 백업 플러그인
- 동기화 플러그인
- 테마 엔진
- 입력기
- 단축키 매니저
- 알림 필터
- 클라우드 스토리지 연동 플러그인
```

이 유형의 애드온은 코어 시스템을 직접 대체해서는 안 된다. 대신 코어 시스템이 제공하는 공식 API 위에서 동작해야 한다.

---

## 6.4 위젯 및 패널형 애드온

데스크톱, 대시보드, 사이드바, 상태바 등에 붙는 작은 기능 단위다.

예시는 다음과 같다.

```text
- 날씨 위젯
- 시계 위젯
- 메모 위젯
- 캘린더 위젯
- 시스템 모니터
- 빠른 실행 패널
- 음악 컨트롤러
- 최근 파일 위젯
- 할 일 위젯
- RSS 헤드라인 위젯
```

웹OS에서는 위젯형 애드온이 특히 유용하다. 브라우저 기반 UI와 잘 어울리며, 사용자가 자신의 작업 공간을 개인화할 수 있게 해준다.

---

# 7. 애드온 분류 기준

애드온 여부를 판단할 때는 다음 기준을 사용한다.

| 판단 질문                         | 예시                                | 분류            |
| ----------------------------- | --------------------------------- | ------------- |
| 없으면 OS 부팅, 로그인, 복구, 설정이 어려운가? | 계정, 설정, 권한 관리자                    | 코어 시스템        |
| 없으면 파일 접근이나 시스템 조작이 어려운가?     | 파일 탐색기, 터미널                       | 코어 시스템        |
| 없으면 기본 파일 열기가 불편한가?           | 이미지 뷰어, 영상 플레이어, 텍스트 편집기          | 코어앱           |
| 없어도 OS는 작동하지만 생산성이 높아지는가?     | 마크다운 편집기, 메모, 캘린더                 | 애드온           |
| 특정 사용자군만 주로 사용하는가?            | Git 클라이언트, API 클라이언트, SQLite 브라우저 | 애드온           |
| 여러 대체 앱이 존재할 수 있는가?           | 음악 플레이어, 코드 에디터, PDF 도구           | 애드온 또는 번들 애드온 |
| 시스템 권한이 강한가?                  | 백업, 동기화, 입력기, 자동화 도구              | 시스템 확장형 애드온   |
| 삭제해도 OS가 정상 작동하는가?            | 테마, 위젯, 마크다운 편집기                  | 애드온           |

# 8. 번들 애드온 개념

코어와 애드온 사이에는 애매한 영역이 존재한다.

예를 들어 마크다운 편집기, 메모 앱, PDF 뷰어, 압축 관리자, 음악 플레이어는 없어도 OS가 망가지지는 않는다. 하지만 기본 제공하면 사용자 경험이 좋아진다.

이런 앱은 **번들 애드온**으로 분류하는 것이 좋다.

## 8.1 번들 애드온 정의

**번들 애드온**은 기본 설치에 포함되지만, 구조적으로는 애드온이며 사용자가 삭제하거나 교체할 수 있는 앱이다.

```text
- 기본 제공된다.
- 삭제 가능하다.
- 코어 시스템에 직접 포함되지 않는다.
- OS 업데이트와 분리해서 업데이트할 수 있다.
- 사용자가 다른 앱으로 대체할 수 있다.
```

---

## 8.2 번들 애드온 후보

```text
- 마크다운 편집기
- 메모 앱
- PDF 뷰어 또는 PDF 주석 도구
- 압축 관리자
- 음악 플레이어
- 코드 에디터 라이트
- 테마 관리자
- 스크린샷 도구
- 클립보드 관리자
```

번들 애드온 개념을 도입하면 “이 앱은 코어인가, 애드온인가?”라는 애매한 문제를 깔끔하게 해결할 수 있다.

---

# 9. 권한과 보안 기준

애드온은 코어 시스템보다 자유롭게 확장될 수 있어야 하지만, 그만큼 권한 관리가 중요하다.

애드온은 필요한 권한만 요청해야 하며, 사용자는 해당 권한이 어떤 의미인지 명확히 알 수 있어야 한다.

예를 들어 마크다운 편집기는 사용자가 선택한 파일만 읽고 쓰면 충분하다. 반면 백업 도구나 동기화 도구는 넓은 파일 접근 권한과 백그라운드 실행 권한이 필요할 수 있다.

따라서 애드온 권한은 세분화해야 한다.

```text
- 파일 열기 권한
- 파일 저장 권한
- 전체 파일 접근 권한
- 네트워크 요청 권한
- 계정 정보 접근 권한
- 클립보드 읽기/쓰기 권한
- 알림 생성 권한
- 백그라운드 실행 권한
- 화면 캡처 권한
- 터미널 또는 프로세스 실행 권한
- 시스템 설정 읽기/쓰기 권한
```

특히 다음 권한은 고위험 권한으로 관리해야 한다.

```text
- 전체 파일 접근
- 계정 토큰 접근
- 백그라운드 실행
- 화면 캡처
- 터미널 실행
- 시스템 설정 변경
- 자동화 스크립트 실행
```

애드온은 권한을 한 번에 모두 요구하기보다, 기능을 사용하는 순간 필요한 권한을 요청하는 방식이 바람직하다.

---

# 10. 애드온 등급

애드온은 권한 수준에 따라 등급을 나누는 것이 좋다.

## Level 1. 일반 앱

낮은 권한으로 동작하는 독립 앱이다.

```text
- 계산기
- 메모 앱
- 마크다운 편집기
- 음악 플레이어
- 이미지 편집기
```

주요 권한:

```text
- 자신의 앱 데이터 접근
- 사용자가 선택한 파일 접근
- 제한된 네트워크 접근
```

---

## Level 2. 파일 핸들러

특정 파일 형식을 열고 저장할 수 있는 앱이다.

```text
- PDF 뷰어
- CSV 뷰어
- EPUB 리더
- JSON 뷰어
- 마크다운 편집기
```

주요 권한:

```text
- 연결된 파일 열기
- 사용자가 허용한 파일 저장
- 파일 탐색기와의 연결
```

---

## Level 3. 시스템 연동 애드온

시스템 UI나 상태와 연결되는 애드온이다.

```text
- 클립보드 관리자
- 스크린샷 도구
- 런처 확장
- 검색 플러그인
- 위젯
```

주요 권한:

```text
- 클립보드 접근
- 화면 캡처
- 알림 생성
- 시스템 상태 읽기
```

---

## Level 4. 고권한 시스템 확장

시스템에 깊게 연결되는 애드온이다.

```text
- 백업 도구
- 동기화 도구
- 클라우드 드라이브 마운트
- 자동화 도구
- 입력기
- 보안 도구
```

주요 권한:

```text
- 넓은 파일 접근
- 백그라운드 실행
- 네트워크 지속 연결
- 계정 연동
- 시스템 이벤트 구독
```

이 등급의 애드온은 별도 승인, 서명, 검증 절차를 두는 것이 좋다.

# 11. 설치, 비활성화, 삭제 원칙

애드온은 설치뿐 아니라 비활성화와 삭제까지 안전해야 한다.

애드온의 생명주기는 다음과 같이 설계한다.

```text
1. 검색
2. 설치
3. 권한 승인
4. 활성화
5. 업데이트
6. 비활성화
7. 삭제
8. 데이터 정리 또는 보존 선택
```

애드온을 삭제할 때는 사용자에게 다음 선택지를 제공하는 것이 좋다.

```text
- 앱만 삭제
- 앱과 설정 삭제
- 앱, 설정, 사용자 데이터 모두 삭제
```

이렇게 해야 사용자가 실수로 중요한 데이터를 잃지 않으면서도 시스템을 깨끗하게 관리할 수 있다.

# 13. 최종 분류 기준 요약

웹OS의 각 계층은 다음 기준으로 구분한다.

## 코어 시스템

> 없으면 OS를 사용할 수 없거나, 복구할 수 없거나, 보안과 안정성이 크게 흔들리는 기능.

## 코어앱

> OS 설치 직후 기본적인 파일과 콘텐츠를 열고, 보고, 간단히 편집할 수 있게 하는 기본 앱.

## 번들 애드온

> 기본 제공하면 편리하지만, 삭제되거나 교체되어도 OS가 망가지지 않는 선택형 앱.

## 애드온

> 사용자의 목적, 취향, 직업, 작업 방식에 따라 설치하는 확장 앱 또는 기능.

---

# 14. 최종 철학 문장

웹OS의 구성 철학은 다음과 같이 정리할 수 있다.

> **코어 시스템은 OS의 생존과 안정성을 책임진다. 코어앱은 기본 사용성을 보장한다. 애드온은 사용자의 목적과 취향에 따라 작업 환경을 확장한다.**

그리고 애드온의 철학은 다음과 같이 정의할 수 있다.

> **애드온은 코어를 대체하지 않고, 코어 위에서 사용자의 작업 세계를 확장하는 선택형 기능이다. 애드온은 안전하게 설치, 비활성화, 삭제될 수 있어야 하며, 시스템 안정성과 사용자 데이터의 소유권을 침해하지 않아야 한다.**

---

# 15. 결론

웹OS는 코어 시스템, 코어앱, 애드온을 명확히 분리함으로써 안정성과 확장성을 동시에 확보할 수 있다.

코어 시스템은 작고 견고해야 한다. 코어앱은 기본 파일 조작과 콘텐츠 소비를 가능하게 해야 한다. 애드온은 사용자의 필요에 따라 자유롭게 확장되되, 코어 시스템을 침범하거나 불안정하게 만들어서는 안 된다.

특히 **번들 애드온** 개념을 도입하면 기본 제공 기능과 선택형 기능 사이의 애매한 경계를 유연하게 처리할 수 있다. 마크다운 편집기, 메모 앱, PDF 도구, 압축 관리자, 스크린샷 도구, 클립보드 관리자처럼 유용하지만 필수는 아닌 기능은 번들 애드온으로 제공하는 것이 적절하다.

최종적으로 웹OS는 다음 구조를 지향한다.

```text
Core System
  OS의 생존, 보안, 설정, 계정, 파일 접근, 복구

Core Apps
  기본 파일 열기, 보기, 재생, 간단 편집

Bundled Add-ons
  기본 제공되지만 삭제 가능한 편의 기능

Add-ons
  사용자의 작업 방식에 따라 설치하는 선택형 확장 기능
```

이 구조를 따르면 웹OS는 기본적으로 안정적이면서도, 사용자에 따라 가볍게도 강력하게도 확장될 수 있는 운영체제가 된다.
