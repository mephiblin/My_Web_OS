# My Web OS 기획서

## 1. 프로젝트 정의

`My Web OS`는 기존 Windows, macOS, Linux 같은 로컬 운영체제를 대체하려는 진짜 OS가 아니다.

이 프로젝트는 이미 존재하는 로컬 PC 환경을 웹 브라우저 기반 UI로 감싸고, 그 위에 더 편리한 작업 화면, 패키지 센터, 로컬 접근 기능, 운영 상태 확인, 백업/복구, 커스터마이징 기능을 얹는 개인용 웹 작업환경 레이어다.

즉, 목표는 커널, 드라이버, 프로세스 스케줄러, 완전한 멀티유저 권한 모델을 새로 만드는 것이 아니다. 목표는 사용자가 매일 쓰는 PC의 파일, 터미널, 실행 환경, 시스템 상태, 앱 관리 흐름을 웹 UI 안에서 더 쉽게 보고, 다루고, 확장할 수 있게 만드는 것이다.

한 문장으로 정의하면 다음과 같다.

> My Web OS는 로컬 PC를 웹 기반 작업환경으로 래핑하여, 기존 OS 위에서 더 자유로운 커스터마이징, 패키지 부착, 운영 편의성, 복구 가능성을 제공하는 개인용 Web OS 레이어다.

## 2. 핵심 관점

이 프로젝트에서 말하는 `Web OS`는 엄밀한 의미의 운영체제가 아니다. 여기서의 OS는 사용자가 체감하는 작업환경, 조작면, 앱 실행 표면, 상태 관리 체계를 의미한다.

기존 OS는 그대로 존재한다. My Web OS는 그 위에 올라가는 웹 기반 조작면이다. 사용자는 브라우저를 통해 로컬 PC의 주요 기능에 접근하고, 자신에게 필요한 도구를 패키지처럼 만들고 붙이며, 데스크톱처럼 창과 앱을 다룬다.

따라서 이 프로젝트는 다음 중간 지점을 목표로 한다.

- 단순한 웹 데스크톱 장난감은 아니다.
- 진지한 수준의 독립 운영체제도 아니다.
- 로컬 PC를 더 편하게 다루기 위한 웹 기반 개인 작업대다.
- DSM처럼 상태와 기능을 한눈에 볼 수 있는 관리 표면을 지향한다.
- 기존 OS보다 더 자유로운 메인 화면 커스터마이징과 기능 부착을 지향한다.

## 3. 프로젝트 철학

### 3.1 로컬 환경을 숨기지 않고 감싼다

이 프로젝트의 출발점은 로컬 PC다. 파일, 터미널, 시스템 상태, 실행 중인 프로세스, 로컬 저장소는 프로젝트의 중요한 기반이다.

다만 이 접근을 날것 그대로 노출하는 것이 아니라, 웹 UI와 API를 통해 사용자가 이해하기 쉬운 형태로 감싼다. 로컬 접근성은 위험해서 감출 대상이 아니라, 명확한 경계와 복구 수단 안에서 잘 드러내야 하는 핵심 가치다.

### 3.2 기존 OS의 제약을 넘어서는 조작면을 만든다

Windows나 macOS는 강력하지만 사용자가 바꿀 수 있는 표면이 제한되어 있다. 바탕화면, 앱 런처, 창 관리, 설정, 앱 설치 방식은 OS가 정한 방식 안에서 움직인다.

My Web OS는 프론트엔드 메인 화면을 중심으로, 사용자가 자신의 작업환경을 더 자유롭게 바꾸는 것을 목표로 한다.

여기서 커스터마이징은 단순히 배경화면이나 색을 바꾸는 것이 아니다. 필요한 도구를 만들고, 패키지처럼 붙이고, 화면 배치를 바꾸고, 로컬 기능과 연결하며, 자기만의 작업 흐름을 구성하는 것을 뜻한다.

### 3.3 패키지 센터는 앱스토어가 아니라 제작소이자 정비소다

패키지 센터는 단순히 앱 목록을 보여주고 설치하는 곳이 아니다.

패키지 센터는 다음 역할을 함께 가져야 한다.

- 필요한 기능을 새 패키지로 만든다.
- 외부 또는 로컬 패키지를 설치한다.
- 설치된 패키지의 상태를 본다.
- 패키지의 런타임, 로그, 이벤트, 헬스 상태를 확인한다.
- 업데이트 정책과 채널을 관리한다.
- 문제가 생기면 백업, 복구, 롤백을 수행한다.

따라서 패키지 센터는 스토어이면서 동시에 운영 콘솔이고, 개발자 모드의 시작점이며, 사용자 확장의 중심이다.

### 3.4 운영 신뢰성은 과하지 않게, 그러나 실제로 필요하다

이 프로젝트는 엔터프라이즈급 운영체제나 대규모 서버 관리 도구를 목표로 하지 않는다. 하지만 로컬 파일, 터미널, 패키지 설치, 런타임 실행을 다루기 때문에 최소한의 운영 신뢰성은 반드시 필요하다.

여기서 말하는 운영 신뢰성은 거대한 보안 체계를 뜻하지 않는다. 사용자가 직접 쓰는 개인용 도구로서 다음을 보장하는 수준이다.

- 실패했을 때 원인을 알 수 있다.
- 작업 상태가 로그와 이벤트로 남는다.
- 설치나 업데이트 전에 백업할 수 있다.
- 문제가 생기면 롤백할 수 있다.
- 패키지를 만들거나 설치하기 전에 기본 검증을 수행할 수 있다.
- 위험한 동작은 명시적인 흐름과 권한 안에서 실행된다.

### 3.5 Web OS는 기존 OS 위의 증강 레이어다

이 프로젝트는 기존 OS를 없애지 않는다. 오히려 기존 OS의 능력을 활용한다.

로컬 OS가 파일 시스템, 프로세스, 네트워크, 런타임을 제공하고, My Web OS는 그 위에서 사용자가 더 쉽게 다룰 수 있는 웹 기반 인터페이스와 확장 시스템을 제공한다.

즉, My Web OS는 OS 대체재가 아니라 OS 증강 레이어다.

## 4. 파일스테이션 Places 기본 연결 정책

설치 직후(Windows/Ubuntu) 사용자 접근성이 떨어지지 않도록, File Station의 `Places`는 OS 표준 사용자 디렉터리를 기본으로 자동 연결한다.

### 4.1 기본 원칙

- 사용자는 설치 직후 바로 `Desktop`, `Documents`, `Downloads`에 접근 가능해야 한다.
- `Home`은 Desktop이 아니라 사용자 홈 루트(예: `C:\Users\<user>`, `/home/<user>`)를 의미한다.
- `Desktop`은 Home 하위의 작업 표면으로 `Places`에 별도 항목으로 노출한다.
- 보안 경계는 유지하되, 기본 허용 루트는 사용자 작업 폴더 중심으로 제공한다.

### 4.2 기본 Places 구성

- Windows:
  - Home
  - Desktop
  - Documents
  - Downloads
  - Pictures
  - Music
  - Videos
- Ubuntu:
  - Home
  - Desktop
  - Documents
  - Downloads
  - Pictures
  - Music
  - Videos

### 4.3 실행 정책

- `ALLOWED_ROOTS`가 비어 있거나 레거시 제한값(프로젝트 루트 단일 경로)인 경우, 위 Places 경로를 자동 기본값으로 채운다.
- `INITIAL_PATH`는 기본적으로 `Documents`를 우선 사용하고, 없으면 `Home`으로 폴백한다.
- Places에 표시된 항목은 실제 탐색 가능한 경로와 일치해야 하며, 표시-접근 불일치(클릭 시 403)는 제품 결함으로 본다.

## 4. 비목표

다음은 이 프로젝트가 직접 목표로 삼지 않는 영역이다.

- 커널 수준 운영체제 구현
- 드라이버, 스케줄러, 메모리 관리자 구현
- Windows/macOS/Linux 대체
- 완전한 컨테이너 격리 또는 강력한 멀티테넌트 보안
- 엔터프라이즈급 장비 관리 시스템
- 불특정 다수에게 공개되는 SaaS형 원격 관리 플랫폼
- 실제 운영 장비를 무제한으로 제어하는 고위험 자동화 도구

이 프로젝트는 개인용, 로컬 중심, 실험과 생산성을 위한 웹 작업환경이다.

## 5. 사용자 경험 목표

사용자는 브라우저를 열었을 때 하나의 웹 데스크톱을 만난다. 이 데스크톱은 단순한 모양 흉내가 아니라 실제 로컬 PC와 연결된 작업 공간이어야 한다.

사용자는 다음을 기대할 수 있어야 한다.

- 파일을 보고 관리할 수 있다.
- 터미널을 열어 로컬 명령을 실행할 수 있다.
- 시스템 상태를 확인할 수 있다.
- 앱을 창 형태로 열고 닫을 수 있다.
- 필요한 도구를 패키지로 설치하거나 직접 만들 수 있다.
- 설치된 패키지의 상태, 로그, 이벤트, 백업, 롤백 정보를 볼 수 있다.
- 메인 화면과 작업 흐름을 자신의 목적에 맞게 바꿀 수 있다.

좋은 사용 경험은 "진짜 OS처럼 완벽하다"가 아니라 "내 PC를 더 편하게 다루게 해준다"에 가깝다.

## 6. Synology DSM과의 관계

이 프로젝트는 Synology DSM과 닮은 점이 있다. DSM은 NAS의 상태, 파일, 패키지, 서비스, 설정을 웹 UI로 통합해 보여준다. 사용자는 브라우저에서 장치를 운영하고 필요한 패키지를 설치한다.

My Web OS도 비슷하게 다음 가치를 지향한다.

- 모든 상태를 한눈에 볼 수 있다.
- 로컬 자원에 접근할 수 있다.
- 필요한 기능을 패키지로 붙일 수 있다.
- 설정, 로그, 상태, 복구 흐름이 UI 안에 있다.

다만 DSM이 NAS 운영 콘솔이라면, My Web OS는 로컬 PC의 개인 작업환경에 더 가깝다. NAS 관리보다는 사용자의 데스크톱, 파일, 앱, 도구, 자동화 흐름을 웹으로 묶는 데 초점이 있다.

## 7. 구조적 개념

### 7.1 Host Layer

Host Layer는 실제 로컬 PC와 연결되는 층이다.

주요 역할은 다음과 같다.

- 로컬 파일 시스템 접근
- 터미널 실행
- 시스템 메트릭 조회
- 로그와 감사 기록
- 로컬 설정 관리
- 로컬 런타임과 프로세스 제어

이 층은 강력하기 때문에 사용자가 무엇을 실행하고 있는지 알 수 있어야 한다.

### 7.2 Web Desktop Layer

Web Desktop Layer는 사용자가 직접 마주하는 프론트엔드 작업환경이다.

주요 역할은 다음과 같다.

- 바탕화면
- 창 관리
- 앱 아이콘
- 작업 표시줄
- 위젯
- 알림
- 설정 화면
- 파일 탐색기
- 패키지 센터

이 층은 프로젝트의 정체성을 가장 강하게 드러낸다. 기존 OS보다 자유로운 메인 화면과 작업 흐름을 제공해야 한다.

### 7.3 Sandbox / Package Layer

Sandbox / Package Layer는 사용자가 추가하는 기능을 패키지 형태로 관리하는 층이다.

주요 역할은 다음과 같다.

- 패키지 설치
- 패키지 생성
- 패키지 import/export
- 런타임 설정
- 의존성 검사
- 채널 기반 업데이트 정책
- 헬스 체크
- 백업과 롤백
- 템플릿 품질 게이트

이 층은 프로젝트가 단순 데스크톱 UI를 넘어 확장 가능한 작업환경이 되는 핵심이다.

## 8. 패키지 센터의 제품 방향

패키지 센터는 이 프로젝트의 가장 중요한 축 중 하나다.

초기에는 앱을 설치하고 제거하는 화면처럼 보일 수 있지만, 장기적으로는 사용자가 자신의 Web OS를 확장하는 중심 도구가 되어야 한다.

패키지 센터는 다음 방향을 가진다.

- Store 탭은 설치 가능한 패키지와 템플릿을 보여준다.
- Installed 탭은 설치된 패키지의 운영 상태를 보여준다.
- 패키지는 단순 HTML 앱뿐 아니라 서비스, 위젯, 도구, 자동화 단위까지 확장될 수 있다.
- 패키지 생성은 사용자가 자기 작업환경을 직접 확장하는 기본 흐름이 된다.
- 업데이트는 무조건 최신으로 올리는 것이 아니라 채널과 안정성 정책에 따라 제어된다.
- 실패한 설치나 업데이트는 백업과 롤백으로 회복 가능해야 한다.

패키지 센터는 "앱을 받는 곳"이 아니라 "내 작업환경을 조립하는 곳"이다.

## 9. 커스터마이징의 의미

이 프로젝트에서 커스터마이징은 장식이 아니다.

커스터마이징은 사용자가 로컬 PC 위에 자기만의 작업 방식을 만드는 능력이다.

예시는 다음과 같다.

- 자주 쓰는 파일 작업을 패키지로 만든다.
- 로컬 로그를 보는 전용 대시보드를 만든다.
- 다운로드, 사진 정리, 백업, 미디어 관리 도구를 붙인다.
- 시스템 상태를 위젯으로 배치한다.
- 개발용 터미널, 문서 편집기, 로컬 서버 관리 도구를 하나의 작업면에 둔다.
- 메인 화면을 사용자의 목적에 맞게 재구성한다.

기존 OS는 어느 정도만 바꿀 수 있다. My Web OS는 웹 프론트엔드라는 특성을 이용해 사용자가 훨씬 더 많은 표면을 바꿀 수 있게 한다.

## 10. 안전성과 자유도의 균형

이 프로젝트는 로컬 접근과 커스터마이징을 중요하게 본다. 하지만 자유도가 커질수록 실수와 실패 가능성도 커진다.

따라서 다음 균형이 필요하다.

- 사용자가 원하는 기능을 만들 수 있어야 한다.
- 위험한 권한은 명시적으로 드러나야 한다.
- 패키지 설치 전 기본 검증이 있어야 한다.
- 업데이트 전 백업이 가능해야 한다.
- 실패 후 복구와 롤백이 가능해야 한다.
- 로그와 이벤트를 통해 무슨 일이 있었는지 추적할 수 있어야 한다.

이 안전성은 프로젝트를 무겁게 만들기 위한 것이 아니라, 사용자가 더 안심하고 실험할 수 있게 하기 위한 기반이다.

## 11. 성공 기준

이 프로젝트가 좋은 방향으로 가고 있는지는 다음 질문으로 판단한다.

- 사용자가 브라우저에서 실제 로컬 PC 작업을 더 쉽게 할 수 있는가?
- 웹 데스크톱이 단순 장식이 아니라 실제 작업 흐름을 줄여주는가?
- 패키지를 만들고 붙이는 과정이 충분히 가벼운가?
- 설치된 패키지의 상태와 문제를 사용자가 이해할 수 있는가?
- 실패했을 때 되돌릴 수 있는가?
- 기존 OS보다 더 자유롭게 작업환경을 바꿀 수 있는가?
- 개인용 도구로서 너무 무겁거나 복잡해지지 않았는가?

## 12. 프로젝트 톤

이 프로젝트는 과도하게 무거운 OS 연구 프로젝트가 아니다. 그렇다고 단순한 데스크톱 스킨도 아니다.

적절한 톤은 다음과 같다.

> 로컬 PC를 더 편하게 쓰기 위한, 웹 기반 개인 작업대.

이 톤을 유지하는 것이 중요하다. 기능을 추가할 때마다 "이게 진짜 OS다운가?"보다 "이게 내 로컬 PC를 더 잘 다루게 해주는가?"를 먼저 물어야 한다.

## 13. 앞으로의 검토 기준

프로젝트 전체 검토를 할 때는 일반적인 코드 품질만 보면 부족하다. 이 프로젝트의 목적에 맞게 다음 기준으로 봐야 한다.

1. 로컬 PC 래핑이 실제 사용 흐름에 도움이 되는가
2. 웹 데스크톱 UI가 작업 공간으로 충분히 자연스러운가
3. 패키지 센터가 설치 화면을 넘어 확장과 정비의 중심이 되는가
4. Host Layer와 Sandbox / Package Layer의 경계가 적절한가
5. 실패 상태가 사용자에게 충분히 드러나는가
6. 백업, 복구, 롤백 흐름이 실제로 동작하는가
7. 커스터마이징 자유도가 프로젝트의 핵심 가치로 살아 있는가
8. 개인용 도구로서 복잡도가 과도하게 커지지 않았는가

## 14. 최종 정리

My Web OS는 "브라우저로 만든 가짜 OS"가 아니다. 또한 "기존 운영체제를 대체하는 진짜 OS"도 아니다.

이 프로젝트는 기존 로컬 OS 위에 올라가는 웹 기반 작업환경이다. 로컬 PC의 기능을 웹 UI로 감싸고, 사용자가 필요한 기능을 패키지처럼 붙이며, 기존 OS보다 더 자유롭게 작업면을 바꿀 수 있게 만드는 것이 핵심이다.

따라서 이 프로젝트의 본질은 다음과 같다.

> 기존 PC 환경을 웹으로 래핑하고, 그 위에 더 편한 조작면, 더 높은 커스터마이징 자유도, 패키지 기반 확장, 기본적인 운영 신뢰성을 더하는 개인용 Web OS 레이어.

## 15. 코드 검토 기반 현재 상태와 실행 과제

검토일: 2026-04-21

이 절은 현재 코드 기준으로 `기획서.md`의 철학이 어느 정도 구현되어 있는지, 그리고 이 문서의 방향대로 가기 위해 무엇을 수정, 추가, 변경해야 하는지를 정리한다.

### 15.1 현재 구현 상태 요약

현재 프로젝트는 이미 "브라우저 안에서 로컬 PC를 다루는 웹 작업환경"의 기본 골격을 갖추고 있다.

- 프론트엔드는 Svelte 5 기반이며 `client/src/core/Desktop.svelte`, `Window.svelte`, `Taskbar.svelte`, `Spotlight.svelte`를 중심으로 데스크톱, 창, 작업표시줄, 검색, 알림, 위젯, Agent 레이어가 구성되어 있다.
- 백엔드는 Express + Socket.io 기반이며 `server/index.js`에서 파일, 시스템, 인증, Docker, 설정, Cloud, 미디어, 로그, 패키지, 공유, 서비스, 샌드박스, 런타임 API를 라우팅한다.
- Host Layer 기능은 `server/routes/fs.js`, `system.js`, `docker.js`, `cloud.js`, `logs.js`, `settings.js` 등에 분리되어 있고, 파일 탐색기, 터미널, 리소스 모니터, Docker Manager, Log Viewer, Settings, Control Panel 앱으로 노출된다.
- Sandbox / Package Layer는 `server/storage/inventory/` 구조, `server/routes/sandbox.js`, `server/routes/packages.js`, `server/services/packageRegistryService.js`를 통해 앱 인벤토리, 샌드박스 정적 앱, 앱별 데이터 API, 레지스트리 설치 흐름을 제공한다.
- Service Runtime Layer는 `server/services/runtimeManager.js`, `processSupervisor.js`, `runtimeStateStore.js`, `runtimeProfiles.js`, `server/routes/runtime.js`로 구현되어 `process-node`, `process-python`, `binary`, `sandbox-html` 런타임 구분과 start/stop/restart/logs/events/recover/validate API를 제공한다.
- 운영 신뢰성 쪽은 `packageLifecycleService.js`, `channelUpdatePolicyService.js`, `templateQualityGate.js`가 추가되어 lifecycle, channel, SemVer range, backup, rollback, health, template quality gate가 코드상으로 들어와 있다.
- `PackageCenter.svelte`는 Store/Installed 분리, 레지스트리 소스 관리, 설치/제거, 런타임 제어, 로그, 이벤트, health, lifecycle, channel, backup, rollback, template scaffold UI까지 확장되어 단순 앱스토어보다 운영 콘솔에 가까워지고 있다.

따라서 현재 상태는 "가짜 OS UI" 단계를 넘어섰고, 로컬 PC 래핑과 패키지 운영 기반을 실제로 구현한 단계다. 다만 제품 완성도 관점에서는 아직 여러 기능이 연결은 되었지만 사용자 흐름, 검증, 복구, 보안 경계, 설정 모델이 균일하게 정리되지 않은 상태다.

### 15.2 철학 대비 부족한 부분

현재 코드가 가장 보강해야 할 부분은 "기능 존재"보다 "사용자가 실패를 이해하고 되돌릴 수 있는 흐름"이다.

- Host Layer와 Sandbox Layer의 경계가 개념적으로는 분리되어 있지만, 파일 접근, 인벤토리 보호, 패키지 권한 승인, 위험 명령 승인 흐름이 아직 일관된 UX로 묶이지 않았다.
- Package Center는 운영 콘솔화가 상당히 진행되었지만, 설치 전 권한/복구/호환성 리뷰를 사용자가 쉽게 이해하는 설치 마법사 형태로 제공하지는 않는다.
- 런타임, lifecycle, health, backup, rollback API는 존재하지만 사용자 입장에서는 "무슨 문제가 있고 어떤 조치를 하면 되는지"를 한 화면에서 읽는 진단 경험이 더 필요하다.
- 설정은 `server/config/serverConfig.js`, `defaults.json`, `systemStore.js`, Control Panel, Settings로 나뉘어 있으며 서버 설정과 UI 설정의 역할 구분은 되어 있지만 정보 구조와 경고 문구가 더 명확해야 한다.
- Agent는 화면 우하단 캐릭터/말풍선 형태는 있으나 실제 OS 작업 보조, 명령 승인, 로그 요약, 패키지 문제 해결까지 연결되지는 않았다.
- 테스트 파일은 runtime/lifecycle/channel/template/recovery 계열이 생겼지만 루트 `package.json`의 `npm test`는 아직 테스트 실행기로 연결되어 있지 않다.
- 문서상 핵심 기준인 `Readme-3,md`, `doc/계획표2.md`는 현재 워크트리에서 삭제 상태로 보이며, 최신 source of truth가 흩어질 수 있다.

### 15.3 수정해야 할 것

- `Package Center`의 설치/업데이트 흐름을 "권한 확인 -> 품질 게이트 -> 백업 여부 -> 설치/업데이트 -> 결과/복구 제안" 순서로 정리한다.
- Host/Sandbox 경계 위반 가능성이 있는 API 응답에 명확한 에러 코드와 사용자용 설명을 붙인다. 예: `FS_SYSTEM_PROTECTED`, `PACKAGE_PERMISSION_REVIEW_REQUIRED`, `RUNTIME_COMMAND_NOT_ALLOWED`.
- `Settings`와 `Control Panel`의 역할을 UI에서도 더 분명하게 나눈다. Settings는 서버/보안/경로 설정, Control Panel은 데스크톱/테마/서비스 상태로 고정한다.
- `server/storage/inventory/system/` 상태 파일에 대한 스키마 검증과 자동 복구 정책을 문서와 코드에서 같은 언어로 맞춘다.
- 런타임 오류, health 실패, rollback 가능 여부를 Package Center Installed 탭에서 더 읽기 쉬운 진단 카드로 보여준다.
- `npm test` 또는 별도 스크립트로 현재 `server/tests/*`를 실행하는 공식 검증 명령을 추가한다.

### 15.4 추가해야 할 것

- 패키지 설치 전 권한 리뷰 모달
- template quality gate 결과 상세 UI
- package health report의 원인/조치 추천 필드
- backup/rollback 실행 전 영향 범위 안내
- Host 파일 접근 요청 승인 흐름
- Agent와 Package Center/Log Viewer/Terminal을 연결하는 최소 작업 요약 API
- 시스템 상태, 서비스 상태, 런타임 상태, 패키지 상태를 한 번에 보는 운영 대시보드
- 문서 source of truth 정리: 삭제된 `Readme-3,md`, `doc/계획표2.md`의 최신 내용이 필요하다면 새 문서로 복원하거나 현재 기획서/next 문서로 흡수한다.

### 15.5 변경해야 할 방향

앞으로의 변경은 새 앱을 많이 늘리는 것보다, 현재 있는 기능을 "실제로 믿고 쓸 수 있는 작업 흐름"으로 묶는 방향이 우선이다.

1. 로컬 파일, 터미널, Docker, 시스템 정보는 Host Layer로 분명히 표시한다.
2. 샌드박스 앱, 패키지 파일, 앱별 데이터는 Sandbox / Package Layer로 분명히 표시한다.
3. Package Center는 Store보다 Installed 운영 콘솔의 밀도를 먼저 높인다.
4. 실패 상태는 로그 파일에만 남기지 말고 UI에서 원인, 영향, 다음 조치를 함께 보여준다.
5. 커스터마이징은 디자인 옵션보다 작업 흐름을 저장하고 재현하는 기능으로 확장한다.
6. AI/Agent 기능은 바로 명령 실행보다 "요약, 설명, 승인, 결과 정리"부터 연결한다.
