# My Web OS 기획서 2

## 1. 문서 목적

`기획서.md`가 프로젝트의 정의와 철학을 정리한 문서라면, 이 문서는 제품의 구체적인 기능, UI 방향, 벤치마크 대상, 그리고 앞으로 어떤 점을 강화해야 하는지를 정리하는 보조 기획서다.

이 문서의 기준은 다음과 같다.

- My Web OS는 진짜 운영체제 대체재가 아니다.
- 기존 로컬 PC 환경을 웹 UI로 래핑하는 개인용 작업환경 레이어다.
- 사용자가 브라우저 안에서 로컬 PC를 더 편하게 다루고, 필요한 기능을 붙이고, 자신의 작업면을 자유롭게 커스터마이징하는 것이 핵심이다.
- 기능은 과하게 엔터프라이즈화하지 않되, 로컬 파일/터미널/패키지를 다루는 만큼 최소한의 운영 신뢰성과 복구 가능성을 가져야 한다.

## 2. UI 핵심 방향: 글래스모피즘 데스크톱

이 프로젝트의 UI 핵심은 글래스모피즘 기반 웹 데스크톱이다.

현재 프론트엔드 구조를 보면 바탕화면, 앱 아이콘, 창, 작업표시줄, 설정 패널, 패키지 센터가 모두 반투명 표면, 흐림 효과, 얇은 경계선, 어두운 배경, 파란 계열 액센트로 구성되어 있다. 이 시각 언어는 단순 장식이 아니라 프로젝트의 정체성을 만든다.

My Web OS는 로컬 PC의 기능을 웹으로 감싸는 프로젝트이기 때문에, UI는 너무 딱딱한 관리자 페이지처럼 보이면 안 된다. 동시에 단순히 예쁜 데모 데스크톱처럼 보여서도 안 된다.

따라서 UI의 방향은 다음과 같다.

- 기본 화면은 데스크톱처럼 보여야 한다.
- 창, 아이콘, 위젯, 작업표시줄은 하나의 작업환경처럼 일관되어야 한다.
- 글래스모피즘은 프로젝트의 기본 시각 언어로 유지한다.
- 반투명 표면과 블러는 사용자가 조절 가능해야 한다.
- 어두운 배경과 고대비 텍스트를 기본으로 하되, 액센트 색상은 설정에서 변경 가능해야 한다.
- 시각 효과는 작업을 방해하지 않고, 로컬 PC를 다루는 도구로서의 신뢰감을 해치지 않아야 한다.

현재 UI에서 유지해야 할 핵심 요소는 다음과 같다.

- Wallpaper: 이미지, 비디오, CSS 그라디언트 배경 지원
- Glass Surface: 창, 패널, 카드, 모달의 반투명 표면
- Accent Blue: 활성 상태, 포커스, 주요 액션의 기본 강조색
- Compact Window UX: 작은 타이틀바, 창 이동/리사이즈/최대화/최소화
- Desktop Grid: 앱과 바로가기 아이콘 배치
- Taskbar: 실행 앱, 검색, 알림, 시간, 설정 접근
- Widget Layer: 데스크톱 위에 얹히는 정보 위젯

## 3. UI 구성 요소 정의

### 3.1 Desktop

Desktop은 사용자가 가장 먼저 만나는 메인 작업면이다.

역할은 다음과 같다.

- 앱 아이콘 표시
- 파일/폴더 바로가기 표시
- 위젯 표시
- 창 호스팅
- 배경화면 표시
- 작업표시줄과 알림 센터 연결
- Spotlight 검색 호출

Desktop은 단순 런처가 아니라 My Web OS의 중심 조작면이다.

### 3.2 Window

Window는 앱을 실제 작업 단위로 보여주는 컨테이너다.

역할은 다음과 같다.

- 앱별 독립 창 표시
- 이동, 리사이즈, 최소화, 최대화
- 활성 창 포커스
- 스냅 영역 표시
- 샌드박스 앱 또는 내장 앱 렌더링

Window UX는 너무 무겁지 않아야 한다. 기존 OS처럼 익숙하지만 웹앱답게 빠르게 열리고 닫혀야 한다.

### 3.3 Taskbar

Taskbar는 실행 중인 앱, 검색, 알림, 시간, 설정 접근을 묶는 하단 조작부다.

목표는 Windows/macOS 작업표시줄을 그대로 복제하는 것이 아니라, 웹 데스크톱에서 자주 쓰는 조작을 작게 모아두는 것이다.

### 3.4 Spotlight

Spotlight는 앱, 기능, 파일, 명령으로 빠르게 이동하기 위한 검색/런처다.

장기적으로는 단순 앱 검색을 넘어 다음 기능까지 확장할 수 있다.

- 앱 실행
- 파일 검색
- 최근 작업 열기
- 패키지 센터 검색
- 시스템 명령 실행
- 자동화 액션 실행

## 4. 핵심 기능 정의

### 4.1 File Station

File Station은 로컬 파일 시스템을 웹 UI로 다루는 핵심 앱이다.

현재 방향은 Synology DSM의 File Station처럼 파일을 보고, 열고, 정리하고, 공유하고, 복구하는 기능을 제공하는 것이다.

기능 정의는 다음과 같다.

- 로컬 디렉터리 탐색
- Home, Documents, Downloads, Pictures, Videos, Music 같은 주요 경로 표시
- Inventory 경로 접근
- 파일/폴더 생성
- 이름 변경
- 삭제 시 휴지통 이동
- 휴지통 보기, 복원, 비우기
- ZIP 압축 해제
- 파일 검색
- 파일 미리보기
- 파일 업로드
- 파일 다운로드
- 데스크톱 바로가기 생성
- 폴더 잠금 표시
- 공유 링크 생성
- Cloud/WebDAV 경로 접근

File Station은 My Web OS에서 가장 중요한 "로컬 PC 래핑" 기능이다. 사용자는 이 앱을 통해 브라우저 안에서 실제 로컬 파일을 다룬다.

### 4.2 WebDAV / Cloud 연결

Cloud 기능은 rclone과 WebDAV를 이용해 원격 저장소를 로컬 작업환경 안으로 끌어오는 기능이다.

기능 정의는 다음과 같다.

- rclone provider 목록 조회
- 원격 저장소 remote 생성
- remote 목록 확인
- remote mount 시도
- WebDAV 연결 추가
- File Station에서 `cloud://` 형태의 가상 경로로 접근
- 원격 파일 목록 조회
- 원격 파일 읽기

이 기능은 단순 클라우드 백업이 아니라, My Web OS가 로컬 PC와 외부 저장소를 하나의 작업면 안에 연결하는 통로다.

### 4.3 Terminal

Terminal은 브라우저 안에서 로컬 셸에 접근하는 기능이다.

기능 정의는 다음과 같다.

- Socket.io 기반 터미널 세션
- `node-pty`가 있을 경우 실제 PTY 연결
- fallback 모드 제공
- 입력, 출력, 리사이즈 이벤트 처리

Terminal은 강력한 기능이므로 권한과 사용 맥락이 명확해야 한다. 이 프로젝트가 개인용 로컬 도구라는 전제를 가장 강하게 드러내는 앱이기도 하다.

### 4.4 Resource Monitor

Resource Monitor는 로컬 PC의 상태를 확인하는 앱이다.

기능 정의는 다음과 같다.

- CPU, 메모리, 디스크, 네트워크 상태 확인
- 시스템 개요 표시
- 운영 상태를 사용자가 즉시 이해할 수 있게 요약
- 장기적으로 서비스 상태, 런타임 상태, 패키지 상태와 연결

이 기능은 DSM의 상태 모니터와 유사한 역할을 한다.

### 4.5 Docker Manager

Docker Manager는 로컬 Docker 컨테이너를 웹 UI에서 관리하는 앱이다.

기능 정의는 다음과 같다.

- Docker 컨테이너 목록 조회
- 컨테이너 시작
- 컨테이너 중지
- 컨테이너 재시작
- 컨테이너 강제 제거
- Docker가 설치되어 있지 않거나 실행 중이 아닐 때 명확한 오류 표시

Docker Manager는 My Web OS가 로컬 개발/홈랩 작업환경으로 확장될 수 있음을 보여주는 기능이다.

장기적으로는 다음 기능을 추가할 수 있다.

- 컨테이너 로그 보기
- 포트/이미지/볼륨 표시 강화
- Compose 프로젝트 단위 관리
- 컨테이너 헬스 상태 표시
- 패키지 센터와 Docker 기반 패키지 연결

### 4.6 Media Player / Image Viewer

Media Player는 이미지, 비디오, 오디오를 여는 통합 뷰어다.

이미지 뷰어 기능 정의는 다음과 같다.

- JPG, PNG, GIF, WebP, SVG 등 이미지 열기
- 이미지 확대/축소
- 같은 폴더의 이전/다음 이미지 이동
- 파일 탐색기에서 이미지 더블클릭 시 열기

비디오/오디오 플레이어 기능 정의는 다음과 같다.

- MP4, WebM, MKV, MOV, AVI 등 비디오 재생
- MP3, WAV, OGG, FLAC, M4A 등 오디오 재생
- 재생/일시정지
- 진행 바
- 볼륨 조절
- 전체화면
- 자막 탐색
- 미디어 메타데이터 조회
- 같은 폴더의 이전/다음 미디어 이동

이 기능은 로컬 PC의 사진/영상/음악을 브라우저 데스크톱 안에서 바로 소비하게 만드는 핵심 편의 기능이다.

### 4.7 Document Viewer

Document Viewer는 문서를 창 안에서 확인하는 앱이다.

현재 기능 정의는 다음과 같다.

- 파일 경로 기반 문서 열기
- 브라우저 iframe 기반 문서 표시
- 다운로드 액션 제공
- PDF 같은 브라우저 지원 문서 형식 우선 처리

장기적으로는 다음 기능을 고려할 수 있다.

- PDF 페이지 이동
- 확대/축소 컨트롤 직접 구현
- 텍스트 검색
- 문서 메타데이터 표시
- Markdown/텍스트 문서와 Code Editor 역할 분리

### 4.8 Model Viewer / FBX Viewer

Model Viewer는 3D 모델 파일을 웹에서 확인하는 앱이다.

현재 기능 정의는 다음과 같다.

- Three.js 기반 3D 렌더링
- GLTF / GLB 로딩
- FBX 로딩
- OBJ 로딩
- OrbitControls 기반 회전/이동/확대
- 모델 자동 중앙 정렬
- 모델 크기에 맞춘 카메라 거리 조정
- 조명 구성
- 애니메이션이 있는 모델 재생/일시정지
- 로딩 상태와 오류 상태 표시

FBX Viewer는 이 앱 안의 중요한 사용 시나리오다. 로컬 PC의 3D 리소스를 별도 네이티브 앱 없이 웹 작업환경에서 빠르게 확인할 수 있게 한다.

장기적으로는 다음 기능을 추가할 수 있다.

- 와이어프레임 보기
- 재질/텍스처 정보 표시
- 애니메이션 클립 선택
- 스크린샷 저장
- 기본 단위/축 표시
- 파일 탐색기와 3D 에셋 폴더 연동

### 4.9 Code Editor

Code Editor는 로컬 파일을 편집하기 위한 앱이다.

기능 정의는 다음과 같다.

- 텍스트/코드 파일 열기
- 로컬 파일 저장
- 구문 강조
- 장기적으로 패키지 개발 모드와 연결

이 앱은 패키지 센터의 "제작소" 방향과 결합될 때 가치가 커진다. 사용자는 패키지를 만들고, manifest를 수정하고, HTML/JS/CSS를 편집할 수 있어야 한다.

### 4.10 Log Viewer

Log Viewer는 시스템과 서비스의 로그를 보는 앱이다.

기능 정의는 다음과 같다.

- 감사 로그 조회
- 시스템 로그 조회
- 로그 레벨 필터링
- 검색
- 운영 상태 추적

이 기능은 운영 신뢰성의 사용자 인터페이스다. 사용자는 문제가 생겼을 때 "어디서 무엇이 실패했는지"를 확인할 수 있어야 한다.

### 4.11 Settings App

Settings App은 서버 환경과 기본 설정을 관리하는 앱이다.

기능 정의는 다음과 같다.

- 서버 포트 설정
- 실행 환경 설정
- JWT secret 설정
- 관리자 계정 설정
- 초기 경로 설정
- 허용 루트 설정
- Cloud Manager 진입

Settings App은 서버 레벨 설정을 다루므로 단순 UI 설정과 구분되어야 한다. 사용자에게 "이 설정은 서버 동작에 영향을 준다"는 맥락이 필요하다.

### 4.12 Control Panel

Control Panel은 사용자가 체감하는 데스크톱 환경과 서비스 상태를 관리하는 앱이다.

기능 정의는 다음과 같다.

- 배경 타입 선택: 그라디언트, 이미지, 비디오
- 배경 프리셋 선택
- 배경 이미지/비디오 업로드
- File Picker를 통한 배경 선택
- 배경 표시 방식 선택: fill, fit, stretch, original, tile
- 글래스모피즘 blur 강도 조절
- 투명도 조절
- 액센트 색상 변경
- 시스템 정보 표시
- 서비스 상태 조회
- 서비스 재시작
- About 화면 표시

Control Panel은 이 프로젝트의 커스터마이징 철학을 직접 보여주는 앱이다. 특히 blur, transparency, accent color는 글래스모피즘 UI의 핵심 조절값이다.

### 4.13 Package Center

Package Center는 My Web OS의 확장 중심이다.

기능 정의는 다음과 같다.

- 설치된 패키지 목록
- 원격 레지스트리 소스 관리
- 원격 패키지 설치
- 패키지 import/export
- 패키지 생성
- 패키지 clone
- 패키지 manifest 조회/수정
- 패키지 파일 목록/읽기/쓰기/삭제
- 런타임 상태 표시
- 런타임 start/stop/restart
- 런타임 로그/이벤트 조회
- 패키지 health check
- lifecycle 조회
- release channel 설정
- backup 생성
- rollback 실행
- template quality gate
- scaffold 전 품질 검사
- SemVer 의존성 판정
- 채널 기반 업데이트 정책

Package Center는 단순 앱스토어가 아니다. 사용자가 자기 작업환경을 조립하고 정비하는 중심이다.

### 4.14 Widget Store

Widget Store는 데스크톱 위젯을 추가하고 관리하는 기능이다.

기능 정의는 다음과 같다.

- 시스템 위젯 추가
- URL 또는 HTML 기반 위젯 추가
- 대시보드형 정보 표시
- 데스크톱 위에 고정되는 작은 도구 제공

장기적으로는 패키지 센터와 위젯 스토어가 하나의 패키지 체계로 묶일 수 있다.

### 4.15 Transfer / Share

Transfer와 Share 기능은 파일 이동, 공유 링크, 다운로드 흐름을 보조하는 기능이다.

기능 정의는 다음과 같다.

- 파일 공유 링크 생성
- 공유 링크 만료 시간 설정
- 공유 다운로드 API
- 업로드 큐 표시
- 장기적으로 전송 작업 상태 표시

이 기능은 로컬 PC를 웹 작업환경으로 열었을 때 필요한 실용 기능이다.

## 5. 패키지 유형 정의

장기적으로 My Web OS의 패키지는 다음 유형으로 확장될 수 있다.

### 5.1 App Package

창 안에서 실행되는 일반 앱이다.

예시는 다음과 같다.

- Markdown Workspace
- To-do
- 다운로드 관리자
- 사진 정리 도구
- 로컬 문서 관리 도구

### 5.2 Widget Package

데스크톱 위에 작게 고정되는 정보 패널이다.

예시는 다음과 같다.

- 시스템 상태 위젯
- 네트워크 상태 위젯
- 시계/날씨 위젯
- Docker 상태 위젯
- 다운로드 진행률 위젯

### 5.3 Service Package

UI보다 백그라운드 실행이 중심인 패키지다.

예시는 다음과 같다.

- 파일 감시 서비스
- 백업 서비스
- 로컬 서버 실행기
- 주기적 정리 작업
- API 프록시

### 5.4 Hybrid Package

UI와 서비스가 함께 있는 패키지다.

예시는 다음과 같다.

- 사진 라이브러리 관리 앱
- 로컬 미디어 서버 관리 앱
- Docker Compose 관리 앱
- WebDAV 동기화 관리 앱

### 5.5 Developer Package

개발자가 앱을 만들고 테스트하기 위한 패키지다.

예시는 다음과 같다.

- manifest 편집기
- 패키지 템플릿 생성기
- 호환성 검사 도구
- 로컬 API 테스트 도구

## 6. 벤치마크 프로젝트

이 프로젝트는 완전히 독립적으로 존재하지 않는다. 이미 여러 웹 OS, 웹 데스크톱, 개인 클라우드, OS 시뮬레이터 프로젝트들이 있다.

My Web OS는 그 프로젝트들을 참고하되, 목표를 정확히 다르게 잡아야 한다.

### 6.1 Puter

링크: https://github.com/HeyPuter/puter

Puter는 오픈소스, 셀프호스팅 가능한 인터넷 OS를 지향한다. README 기준으로 개인 클라우드, 파일/앱/게임 저장소, 웹앱과 게임을 만들고 배포하는 플랫폼, Dropbox/Google Drive/OneDrive 대안, 원격 데스크톱 환경 등을 목표로 한다.

벤치마크할 점은 다음과 같다.

- 개인 클라우드형 웹 OS 경험
- 파일과 앱을 한 공간에 두는 제품 방향
- 앱 스토어와 개발자 플랫폼 방향
- 셀프호스팅 가능성
- 웹 기반 OS라는 명확한 브랜드

My Web OS가 다르게 잡을 점은 다음과 같다.

- Puter가 인터넷/클라우드 OS 쪽에 가깝다면, My Web OS는 로컬 PC 래핑에 더 집중한다.
- My Web OS는 로컬 파일, 로컬 터미널, 로컬 Docker, 로컬 런타임 관리가 더 중요하다.
- 클라우드 대체보다 "내 PC를 웹 작업대로 바꾸는 것"을 우선한다.

### 6.2 ArozOS

링크: https://github.com/tobychui/arozos

ArozOS는 저전력 플랫폼을 위한 웹 데스크톱 운영체제를 지향하며, GitHub 주제에도 NAS, Synology NAS, DSM, home server, network attached storage 같은 방향이 드러난다.

벤치마크할 점은 다음과 같다.

- DSM에 가까운 웹 데스크톱 관리 경험
- NAS/홈서버 친화적 기능
- 파일 관리와 저장소 중심 UI
- 저전력 장치에서 돌아가는 실용성
- 웹 기반 장치 관리 콘솔 방향

My Web OS가 다르게 잡을 점은 다음과 같다.

- ArozOS가 NAS/홈서버 장치 운영에 강하다면, My Web OS는 개인 로컬 PC 작업환경에 더 집중한다.
- My Web OS는 데스크톱 UI, 커스터마이징, 로컬 앱/패키지 제작, Docker/터미널/파일을 한 작업면에 묶는 방향이 중요하다.
- DSM식 투명성은 참고하되, 사용자가 매일 쓰는 PC의 프론트엔드 작업면을 더 자유롭게 바꾸는 것이 차별점이다.

### 6.3 Aurora OS.js

링크: https://github.com/mental-os/Aurora-OS.js

Aurora OS.js는 웹 기반 OS 경험을 해킹 시뮬레이터 게임과 결합한 프로젝트다. README 기준으로 React, Vite, Electron 기반의 고충실도 OS 시뮬레이션, 앱, 터미널, 파일 시스템, 게임적 세계관을 목표로 한다.

벤치마크할 점은 다음과 같다.

- 몰입감 있는 OS 시뮬레이션
- 데스크톱과 앱의 완성도
- 터미널, 파일 시스템, 브라우저, 메일, 캘린더 같은 앱 구성
- OS 자체를 하나의 세계/경험으로 만드는 방향
- 디자인과 분위기의 강한 정체성

My Web OS가 다르게 잡을 점은 다음과 같다.

- Aurora OS.js가 게임/시뮬레이터에 가깝다면, My Web OS는 실제 로컬 PC를 다루는 실용 작업환경이다.
- My Web OS는 세계관보다 로컬 접근, 패키지 관리, 복구, 운영 편의성을 우선한다.
- 재미있는 OS 경험은 참고하되, 실제 파일/터미널/도커/패키지 조작의 신뢰성을 중심에 둔다.

### 6.4 ProzillaOS

링크: https://github.com/prozilla-os/ProzillaOS

ProzillaOS는 Ubuntu Linux와 Windows에서 영감을 받은 React/Vite/TypeScript 기반 웹 운영체제다. 웹 데스크톱, 커스텀 앱, 패키지 구조, 문서화된 개발자 경험이 벤치마크 대상이다.

벤치마크할 점은 다음과 같다.

- 웹 데스크톱 UI 완성도
- React 기반 앱/패키지 구조
- 커스텀 앱 개발 가이드
- 데모와 문서 제공
- 웹 OS 프로젝트로서의 접근성

My Web OS가 다르게 잡을 점은 다음과 같다.

- ProzillaOS가 웹 데스크톱 프론트엔드와 커스텀 앱 구조에 강하다면, My Web OS는 로컬 PC 접근과 운영 콘솔 성격을 더 강하게 가져간다.
- 단순 웹 OS 데모보다 로컬 파일, 터미널, Docker, 패키지 lifecycle을 다루는 실사용 도구를 지향한다.

### 6.5 Origin OS

링크: https://github.com/Mistium/Origin-OS

Origin OS는 커스터마이징 가능한 웹 기반 데스크톱 환경이며, OSL이라는 스크립팅 언어와 앱 개발 경험을 포함한다.

벤치마크할 점은 다음과 같다.

- 강한 커스터마이징 방향
- 웹 데스크톱 환경
- 브라우저 안 파일 시스템
- 사용자 앱 작성/실행 경험
- 스크립팅 언어 기반 확장성

My Web OS가 다르게 잡을 점은 다음과 같다.

- Origin OS가 브라우저/스크립팅 기반 커스텀 OS 환경에 가깝다면, My Web OS는 실제 로컬 PC와 연결된 작업환경이다.
- My Web OS의 확장성은 자체 언어보다 패키지 센터, manifest, sandbox-html, process runtime, Docker, 로컬 API 연결을 중심으로 한다.

## 7. My Web OS가 넘어서고자 하는 축

여기서 "넘어선다"는 말은 모든 면에서 더 크고 완성도 높은 OS를 만든다는 뜻이 아니다.

My Web OS가 넘어서고자 하는 것은 다음 축이다.

### 7.1 로컬 PC 밀착도

많은 웹 OS 프로젝트는 브라우저 안 가상 환경이나 클라우드 저장소 중심이다.

My Web OS는 실제 로컬 PC를 다루는 데 집중한다.

- 실제 로컬 파일
- 실제 로컬 터미널
- 실제 Docker
- 실제 시스템 상태
- 실제 로컬 패키지 저장소
- 실제 백업/롤백

### 7.2 패키지 생명주기

단순히 앱을 추가하는 것이 아니라, 패키지를 만들고, 검증하고, 설치하고, 실행하고, 상태를 보고, 백업하고, 롤백하는 전체 흐름을 갖는다.

차별화 목표는 다음과 같다.

- 패키지 템플릿
- 품질 게이트
- 권한/복구/호환성 점검
- SemVer 의존성 판정
- 채널 기반 업데이트 정책
- 런타임 상태/로그/이벤트
- 백업/롤백

### 7.3 사용자 작업면 커스터마이징

기존 웹 OS들이 데스크톱을 제공하더라도, My Web OS는 메인 화면 자체를 사용자의 작업 방식에 맞게 바꾸는 것을 더 중요하게 본다.

차별화 목표는 다음과 같다.

- 배경, 블러, 투명도, 액센트 색 조절
- 위젯 배치
- 앱 바로가기
- 파일 바로가기
- 패키지 기반 기능 추가
- 로컬 작업 흐름에 맞춘 화면 구성

### 7.4 DSM식 운영 투명성

My Web OS는 DSM처럼 사용자가 상태를 알 수 있어야 한다.

차별화 목표는 다음과 같다.

- 시스템 상태
- 서비스 상태
- Docker 상태
- 패키지 상태
- 런타임 상태
- 로그와 이벤트
- health report
- lifecycle history
- backup/rollback 기록

### 7.5 실용성과 가벼움의 균형

진짜 OS처럼 무겁게 만들지 않는다.

대신 다음 기준을 유지한다.

- 개인용으로 설치하고 실행하기 쉬워야 한다.
- 로컬 PC 작업에 바로 도움이 되어야 한다.
- 기능이 늘어도 전체 UX가 복잡해지지 않아야 한다.
- 실패 시 복구할 수 있어야 한다.
- 사용자가 직접 고치고 확장할 수 있어야 한다.

## 9. 디자인 원칙

### 9.1 관리 도구처럼 정확하게, 데스크톱처럼 자연스럽게

My Web OS는 관리자 페이지처럼 기능이 많지만, 사용자는 데스크톱처럼 자연스럽게 다뤄야 한다.

따라서 UI는 다음 균형을 가져야 한다.

- 정보는 조밀하게 보여준다.
- 버튼과 상태는 명확해야 한다.
- 창과 앱 전환은 가벼워야 한다.
- 불필요한 설명 문구는 줄인다.
- 상태와 오류는 숨기지 않는다.

### 9.2 글래스모피즘은 유지하되 과하게 흐리지 않는다

글래스모피즘은 핵심 스타일이지만, 사용성을 해치면 안 된다.

원칙은 다음과 같다.

- 텍스트 대비를 우선한다.
- 블러 강도는 조절 가능해야 한다.
- 주요 정보 패널은 충분히 읽기 쉬워야 한다.
- 경계선과 그림자로 계층을 구분한다.
- 작업 앱 내부는 장식보다 가독성을 우선한다.

### 9.3 기능 앱은 작고 명확하게

각 앱은 하나의 역할을 분명히 가져야 한다.

- File Station은 파일 관리
- Terminal은 명령 실행
- Docker Manager는 컨테이너 관리
- Package Center는 패키지 생명주기
- Control Panel은 데스크톱 환경 설정
- Settings는 서버 설정
- Media Player는 미디어 소비
- Model Viewer는 3D 파일 확인

역할이 섞이면 사용자는 어디서 무엇을 해야 하는지 헷갈린다.

## 10. 정리

`기획서.md`의 핵심이 "이 프로젝트가 무엇인가"라면, `기획서2.md`의 핵심은 "이 프로젝트가 어떤 기능과 UI로 그 정의를 실현할 것인가"다.

My Web OS는 다음 방향을 가져야 한다.

- 글래스모피즘 기반 웹 데스크톱
- 로컬 PC 기능 래핑
- File Station 중심의 로컬 파일 작업
- Terminal, Docker, Resource Monitor 같은 실제 로컬 도구
- Media/Image/Document/Model Viewer 같은 일상 작업 앱
- Settings와 Control Panel을 통한 환경 조정
- Package Center를 통한 기능 제작, 설치, 운영, 복구
- Puter, ArozOS, Aurora OS.js, ProzillaOS, Origin OS를 벤치마크하되, 로컬 PC 밀착도와 패키지 운영성으로 차별화

최종적으로 이 프로젝트는 "웹으로 만든 운영체제 흉내"가 아니라, 사용자의 로컬 PC를 더 자유롭고 편하게 다루기 위한 웹 기반 개인 작업환경이 되어야 한다.

## 11. 코드 검토 기반 기능별 현재 상태와 보강 계획

검토일: 2026-04-21

이 절은 `기획서2.md`의 기능 정의를 현재 코드와 대조한 결과다. 상태 표기는 다음 기준을 사용한다.

- 구현됨: 기본 사용자 흐름이 코드에 있고 UI/API가 연결되어 있음
- 일부 구현됨: 핵심 골격은 있으나 기획서 수준의 사용 흐름 또는 운영성이 부족함
- 미구현: 현재 코드에서 독립 기능으로 확인되지 않음

### 11.5 우선 수정 항목

P0는 이미 존재하는 기능을 사용자가 믿고 쓸 수 있게 만드는 작업이다.

- Package Center Installed 탭을 운영 콘솔로 완성한다. health, runtime, lifecycle, events, logs, backup, rollback을 카드형 진단 흐름으로 묶는다.
- 패키지 설치 전 리뷰 흐름을 추가한다. 권한, runtime type, dependency, compatibility, quality gate, backup 여부를 설치 전에 보여준다.
- Settings와 Control Panel의 경계를 UI 문구와 탭 구조에서 분명히 한다.
- File Station에서 인벤토리 보호/허용 루트/위험 작업의 에러 메시지를 사용자 친화적으로 정리한다.
- 테스트 실행 명령을 정리한다. 현재 `server/tests/*`는 있으나 루트 `npm test`가 실제 테스트 실행으로 연결되어 있지 않다.

### 11.6 다음 추가 항목

P1은 확장성과 제작 경험이다.

- Package Center에 패키지 생성 마법사, manifest 편집기, 파일 편집기 연결을 추가한다.
- Code Editor에서 패키지 파일을 열 때 Package Center context를 유지한다.
- Widget Store를 Package Center의 package type 체계와 맞춘다.
- Docker Manager에 로그/포트/볼륨/Compose 단위 관리를 추가한다.
- Log Viewer에서 audit log, runtime event, package lifecycle을 하나의 타임라인으로 볼 수 있게 한다.

P2는 로컬 자산 관리와 미디어 경험이다.

- Media Player에 플레이리스트, 백그라운드 오디오, 반복/셔플을 추가한다.
- Document Viewer에 PDF 컨트롤과 텍스트 검색을 추가한다.
- Model Viewer에 와이어프레임, 축 표시, 재질 정보, 스크린샷 저장을 추가한다.
- 사진/영상 정리용 1st-party 패키지를 추가한다.

P3는 DSM/홈랩 방향이다.

- Docker Compose 관리
- WebDAV/cloud 쓰기와 mount 상태 관리
- 백업 작업 관리자
- 서비스/런타임 통합 대시보드
- 다운로드/전송 작업 관리자
