# My Web OS 기획서 4

## 1. 문서 목적

이 문서는 최근 논의된 다음 주제를 정리한다.

- Inventory의 역할 재정의 (시스템 격리 영역)
- 사용자 미디어 관리 영역 신설 (Media Library)
- 바탕화면/음악/영상 파일 흐름의 직관화
- 패키지 앱의 미디어 접근 정책 (raw FS 직접 접근 최소화, 미디어 API 중심)
- 향후 브레인스토밍 후 `AGENTS.md`에 P4로 편입할 실행 메모

## 2. 문제 인식

현재 사용자 체감 문제는 다음과 같다.

1. `Browse from Files`와 `Inventory Gallery`가 같은 곳처럼 보이지만 실제 접근 경계가 다름
2. Inventory는 본래 내부 저장소 의도였지만, 현재는 보안 보호 영역 성격이 강함
3. 사용자 입장에서는 “내 파일을 다루는 공간”과 “시스템이 관리하는 공간”이 UI에서 직관적으로 분리되지 않음

결론: 스토리지 역할을 명확히 이원화해야 한다.

## 3. 핵심 결론

### 3.1 Inventory는 시스템 격리 영역으로 고정

Inventory는 다음을 위한 내부 운영 공간으로 유지한다.

- 패키지 설치 산출물
- 런타임 상태/로그/이벤트/복구 메타데이터
- 시스템 상태 저장 데이터

즉, “사용자 일반 파일 관리 공간”이 아니라 “Web OS 내부 운영 데이터 공간”이다.

### 3.2 사용자용 Media Library를 별도 신설

사용자가 직접 다루는 콘텐츠는 Inventory 밖의 별도 공간으로 분리한다.

예시:

- `storage/media/wallpapers`
- `storage/media/music`
- `storage/media/videos`
- `storage/media/photos`

이 공간은 사용자가 가져오기(import), 정리, 삭제, 교체를 수행하는 운영 대상이다.

## 4. 제품 관점의 분리 모델

### 4.1 설치와 콘텐츠를 분리

- 앱(예: Plex류 앱) 설치/업데이트/런타임 메타: Inventory
- 앱이 읽는 실제 미디어 콘텐츠: Local PC 경로 + Media Library

즉, “앱 자체”와 “앱이 소비하는 데이터”를 분리한다.

### 4.2 앱 접근 방식

- 원칙: 앱은 raw FS를 직접 광범위하게 접근하지 않는다.
- 권장: 앱은 Media API를 통해 접근한다.
- 효과:
  - 권한 스코프 명확화
  - 감사로그 및 정책 적용 용이
  - Host/Sandbox 경계 유지

## 5. Media Library 루트 관리의 의미

`Media Library 루트 관리`는 단순 폴더 지정이 아니라 아래 운영 기능 전체를 뜻한다.

1. 기본 루트 지정 (예: `storage/media`)
2. 카테고리별 서브루트 관리 (`wallpapers/music/videos/photos`)
3. 로컬 폴더 import 정책 (복사/링크/스캔)
4. 앱별 접근 권한 스코프 관리 (`media.read`, `media.write`, 카테고리 범위)
5. 스캔/인덱싱 대상 및 제외 경로 관리
6. UI에서 사용자에게 “현재 라이브러리 위치”와 “가져오기 결과”를 명확히 표시

## 6. UX 원칙 (직관성)

### 6.1 용어 분리

- `Inventory`: System Managed (내부)
- `Media Library`: User Managed (사용자 콘텐츠)
- `Browse Local Files`: 로컬 파일 선택
- `Import to Library`: 라이브러리로 가져오기

### 6.2 추천 사용자 흐름

1. Personalization에서 `Browse Local Files`
2. 파일 선택
3. `Import to Wallpaper Library`
4. 라이브러리 갤러리에서 적용/삭제

핵심은 “직접 참조”보다 “가져오기(Import)”를 기본 모델로 삼는 것이다.

## 7. API/권한 방향

### 7.1 API 계층

- `/api/media-library/*` (목록/검색/메타/삭제/가져오기)
- `/api/media/*` (재생/스트리밍 등 소비 기능)

### 7.2 권한 스코프 초안

- `mediaLibrary.read`
- `mediaLibrary.write`
- `mediaLibrary.import`
- 카테고리 단위 scope (예: wallpapers only)

### 7.3 보안 원칙

- Inventory 직접 노출 금지
- 위험 작업은 명시적 사용자 액션/확인 필요
- 실패 시 명확한 오류 코드 반환

## 8. 단계적 구현 초안

1. 스토리지 구조 분리 (`inventory` vs `media`)
2. Media Library 서비스/라우트 추가
3. Control Panel/Settings에서 라벨과 흐름 정리
4. Browse → Import 모델로 UI 전환
5. 패키지 앱의 미디어 접근을 Media API 중심으로 정리
6. 문서/AGENTS 백로그 업데이트

## 9. Codex 사용량 초기화 이후 작업 메모 (P4 편입 준비)

사용량 초기화 이후 아래 브레인스토밍을 수행한 뒤 `AGENTS.md`에 P4로 편입한다.

### 9.1 브레인스토밍 주제

1. Media Library 정보구조/경로 정책 최종안
2. Import 전략 비교 (복사 vs 링크 vs 가상카탈로그)
3. 앱 권한 모델과 승인 UX
4. Plex류 앱 시나리오 기준 API 계약
5. 인덱싱/성능/대용량 라이브러리 운영 정책

### 9.2 AGENTS.md P4 편입 초안 항목

- Media Library root manager (system/user boundary)
- Browse Local Files -> Import to Library unified flow
- media-library API + permission scopes
- Package apps media access contract (no broad raw FS default)
- Migration guide: existing wallpaper/media references

### 9.3 편입 시 원칙

- P0~P3 안정성 흐름을 훼손하지 않는 범위로 P4 추가
- Host/Sandbox 경계를 약화하지 않음
- UI는 현재 글래스모피즘 언어를 유지하고 용어만 명확화

## 10. 최종 정리

이 기획의 핵심은 다음 한 줄로 요약된다.

> Inventory는 시스템을 위한 격리 공간으로 유지하고, 사용자가 다루는 콘텐츠는 Media Library로 분리하여 운영한다.

이 분리가 완료되면, 사용자 경험은 직관적으로 개선되고, 보안 경계와 운영 일관성도 동시에 강화된다.
