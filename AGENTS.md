This file is the repository-level entry point for coding agents.
Keep it short. Use it as a map, not as the only source of truth.

## Purpose

This project is a browser-based Web OS for managing a local machine.
The target is not a demo desktop UI. The target is a usable web computer environment with:

- host access to local files, terminal, and system metrics
- sandboxed apps with package lifecycle management
- runtime validation, recovery, backup, rollback, and ecosystem growth

Preserve the current visual language and desktop UX while increasing operational depth.

## Architecture Snapshot

- Frontend: Svelte 5 + Vite in `client/`
- Backend: Express + Socket.io in `server/`
- Persistent app inventory: `server/storage/inventory/`
- Runtime state: `server/storage/inventory/system/`
- Package center UI: `client/src/apps/package-center/PackageCenter.svelte`
- Runtime control APIs: `server/routes/runtime.js`
- Package lifecycle APIs: `server/routes/packages.js`

The system is intentionally split between:

- Host layer: local PC access, terminal, files, hardware, settings
- Sandbox layer: packaged apps, isolated app data, managed runtimes

Do not blur these boundaries casually.

## Current Priorities

When in doubt, prioritize work in this order:

1. operational reliability
2. package lifecycle correctness
3. ecosystem tooling for first-party and third-party apps
4. UI integration of runtime and lifecycle data
5. Docker portability

The current highest-priority unfinished work is:

- template quality gate
- Installed-tab runtime/lifecycle console
- SemVer-aware dependency resolution
- channel-based update policy
- recovery and rollback E2E coverage

## Agent Working Rules

- Preserve the existing Glassmorphism desktop UI. Improve flow, density, and clarity without redesigning the product.
- Treat Package Center as an operations console, not only as a store.
- Prefer explicit error codes and recoverable flows over silent failure.
- Prefer small, testable steps over broad refactors.
- Do not revert unrelated user changes.
- Do not commit autogenerated index churn unless the user asks for it.

Important local exception:

- `server/storage/index.json` is frequently autogenerated and should usually stay out of commits.


## Runtime And Package Conventions

- Managed runtime types currently center on `process-node`, `process-python`, and `binary`.
- Sandbox UI apps should remain compatible with `sandbox-html`.
- Lifecycle features already exist for backup, rollback, health checks, logs, events, and validation.
- New package or template work should integrate with lifecycle and QA reporting instead of bypassing it.

If adding a template or app scaffold, plan for:

- permissions review
- recoverability review
- compatibility review
- lifecycle history storage

## Definition Of A Good Change

A strong change in this repository usually does all of the following:

- makes failure states more explicit
- improves recovery behavior or operator visibility
- keeps host vs sandbox boundaries clear
- is validated with `node --check` and, when relevant, `cd client && npm run build`
- updates roadmap docs when behavior or scope materially changes
