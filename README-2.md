# My_Web_OS 프로젝트 전반 분석

## 1) 프로젝트 정체성과 철학

이 프로젝트는 브라우저 안에서 로컬 PC를 제어하는 “웹 기반 운영 환경(Web OS)”을 목표로 합니다. 단순 파일 뷰어가 아니라, **파일/터미널/리소스 모니터링/로그/미디어/설정**을 하나의 데스크톱 UX로 묶어 “로컬 관리 콘솔”로 확장한 것이 핵심입니다. README에서도 개발·실험 단계임을 명시하고, 운영 환경 사용에 대한 강한 경고를 제공합니다.

핵심 철학은 다음과 같이 요약됩니다.
- **통합성**: 여러 관리 기능을 앱 형태로 통합.
- **상태 중심 UX**: 창 상태, 위젯, 설정값을 저장해 OS처럼 연속성 제공.
- **현실 시스템 연동**: node-pty, systeminformation, chokidar, rclone 등 실제 OS/도구와 밀접 결합.
- **실용 우선**: 빠른 구현과 기능 확장을 중시(일부 보안 하드닝은 후순위).

## 2) 시스템 구성(아키텍처)

### 프론트엔드
- Svelte 기반 단일 앱에서 로그인 여부에 따라 `Login`과 `Desktop`을 분기합니다.
- Desktop은 앱 런처, 창 관리자, 위젯 계층, Spotlight, 컨텍스트 메뉴, 태스크바를 총괄합니다.
- 앱 컴포넌트를 ID 기반으로 동적 매핑하여 백엔드 앱 레지스트리와 연결합니다.

### 백엔드
- Express + Socket.io 구조로 HTTP API와 실시간 터미널 스트림을 함께 제공합니다.
- 서비스 초기화(`indexService`, `trashService`, `shareService`) 후 라우터를 모듈별(`/api/fs`, `/api/system`, `/api/auth` 등)로 분리합니다.
- 종료 시 인덱서/PTY 세션을 정리하는 graceful shutdown 로직을 포함합니다.

### 저장소 레이어
- DB 대신 `server/storage` 아래 JSON/로그 파일을 사용하는 **파일 기반 영속화** 구조입니다.
- 앱 목록(`apps.json`), 인덱스(`index.json`), 감사 로그(`audit.log`), 각종 상태(`inventory/state_*.json`)를 직접 관리합니다.

## 3) 코어 모듈 분석

### (A) 창/데스크톱 상태 코어
- `windowStore`는 창 목록/활성 창을 관리하고, debounce 저장으로 서버에 동기화합니다.
- singleton 앱 정책, z-index 포커싱, 가상 데스크톱 연결까지 포함되어 OS형 UX의 핵심 역할을 수행합니다.
- `systemStore`는 테마/배경 상태를 초기화 후 자동 저장하며 CSS 변수로 즉시 반영합니다.

### (B) 파일 시스템 코어
- `pathGuard`는 `ALLOWED_ROOTS` 기준으로 요청 경로를 제한합니다.
- `fs` 라우트는 인증을 기본 적용하고, 검색 시 “인덱스 우선 + 재귀 스캔 폴백” 전략으로 응답성과 정확성을 절충합니다.
- 삭제를 휴지통으로 우회하는 `trashService`는 복구 가능성을 확보해 사용자 안전성을 높입니다.

### (C) 인덱싱 코어
- `indexService`는 chokidar watcher + 큐 기반 배치 업데이트로 이벤트 폭주를 완화합니다.
- 60초 주기 저장, 상대경로 맵 인덱스, 깊이 제한(`INDEX_DEPTH`)으로 성능/범위를 제어합니다.

### (D) 인증/감사 코어
- JWT 인증 미들웨어는 헤더/쿼리 토큰을 모두 지원하며 라우트 단위로 광범위 적용됩니다.
- `auditService`는 JSONL 로그와 크기 기반 로테이션으로 운영 추적성을 확보합니다.

### (E) 터미널 코어
- Socket.io 소켓별 PTY 세션을 맵으로 관리하고, 입력/출력/리사이즈 이벤트를 실시간 전달합니다.
- 연결 종료 시 세션 정리를 수행해 기본적인 리소스 누수 방지를 갖춥니다.

## 4) 강점

1. **기능 통합도가 높음**: 로컬 관리 작업 대부분을 단일 UI로 수행 가능.
2. **확장 구조가 단순명확**: 앱 레지스트리 + 컴포넌트 매핑으로 기능 추가가 쉬움.
3. **실시간성 확보**: 터미널/인덱싱/리소스 모니터링 등 체감 반응이 좋음.
4. **상태 영속화 UX**: 창/설정/위젯 상태를 저장하여 “OS 같은 연속성” 제공.

## 5) 주요 리스크와 기술 부채

1. **보안 하드닝 부족 가능성**
   - 서버에서 `helmet`이 비활성화된 상태.
   - CORS가 현재 전면 허용(`origin: "*"`) 방식.
   - pathGuard는 루트 제한을 제공하지만 심볼릭 링크/마운트 경계 등 고급 위협 모델 검토가 추가로 필요.

2. **운영 신뢰성 측면**
   - 파일 기반 저장은 단일 노드/소규모에는 단순하지만, 동시성·백업·무결성·검색성에서 한계가 존재.
   - 인덱스 메모리 맵 구조는 파일 수가 큰 환경에서 메모리 비용이 증가할 수 있음.

3. **인증 구성 리스크**
   - 관리자 패스워드를 환경변수에서 받아 런타임 해시 생성하는 방식은 단순하지만, 멀티유저/권한 분리 모델로 확장하기 어렵습니다.

## 6) 개선 우선순위 제안

### 0순위(보안)
- `helmet` 적용 복구 및 정책 튜닝.
- CORS allowlist 기반 제어.
- 인증 실패/민감 API rate limit 세분화.

### 1순위(안정성)
- 저장소 레이어를 SQLite/PostgreSQL 등으로 단계적 전환(로그/상태/인덱스 메타).
- 에러 응답 표준화(code/message/details)와 observability(구조화 로그, 메트릭) 강화.

### 2순위(아키텍처)
- 서비스 경계 명확화: 파일/미디어/클라우드/시스템 라우트 별 DTO/유효성 검증 도입.
- 장기적으로 RBAC(역할 기반 권한)와 다중 사용자 모델 설계.

### 3순위(제품성)
- 앱 스토어/위젯 스토어에 서명·검증 흐름 추가.
- 플러그인 생명주기(설치/업데이트/격리) 정의.

## 7) 총평

My_Web_OS는 “브라우저에서 로컬 시스템을 다루는 데스크톱형 플랫폼”이라는 방향성이 매우 분명하고, 이미 코어 사용 시나리오가 유기적으로 연결되어 있습니다. 현재 단계의 핵심 과제는 **보안 하드닝 + 저장소/운영 안정성 강화**이며, 이 두 축을 보완하면 내부 도구/홈랩/개인 클라우드 관리 콘솔로서 경쟁력 있는 기반으로 성장할 수 있습니다.
