# My Web OS 기획서 3

## 1. 문서 목적

이 문서는 My Web OS의 프론트엔드와 UI/UX 방향을 정리한다.

`기획서.md`가 프로젝트의 정의와 철학을 다루고, `기획서2.md`가 기능과 벤치마크를 정리했다면, `기획서3.md`는 사용자가 실제로 마주하는 화면, 커스터마이징 방식, 챗봇 캐릭터, CLI/LLM 경험, 창과 바탕화면의 조작성을 다룬다.

핵심 목표는 다음과 같다.

- 과거 OS와 Web OS들이 말하던 피상적인 커스터마이징을 넘는다.
- 사용자가 시작 메뉴, 작업표시줄, 바탕화면, 창, 창 배경, 패널 구성을 쉽게 바꿀 수 있게 한다.
- 기본 디자인은 현재의 글래스모피즘, 블루 액센트, 어두운 바탕, compact desktop UI를 유지한다.
- 챗봇을 단순 텍스트 입력창이 아니라 캐릭터 기반 인터페이스로 다룬다.
- CLI, DOS풍 터미널, LLM 대화, 말풍선, 채팅창이 하나의 경험으로 이어지게 한다.

## 2. 가짜 커스터마이징에서 벗어나기

많은 OS, Web OS, 데스크톱 시뮬레이터는 커스터마이징을 말하지만 실제로는 제한적이다.

대부분의 커스터마이징은 다음 수준에 머문다.

- 배경화면 변경
- 테마 색상 변경
- 아이콘 크기 변경
- 위젯 몇 개 배치
- 정해진 앱 목록 표시

My Web OS가 원하는 커스터마이징은 이보다 더 깊다.

사용자는 단지 OS의 겉모습을 꾸미는 사람이 아니라, 자신의 로컬 작업환경을 구성하는 사람이어야 한다. 따라서 커스터마이징은 장식이 아니라 작업 흐름을 바꾸는 기능이어야 한다.

이 프로젝트에서 말하는 진짜 커스터마이징은 다음과 같다.

- 시작 메뉴의 구조를 바꿀 수 있다.
- 작업표시줄에 무엇을 둘지 정할 수 있다.
- 바탕화면에 앱, 파일, 폴더, 위젯, 패키지 바로가기를 자유롭게 배치할 수 있다.
- 창의 기본 크기, 위치, 투명도, 배경 스타일을 앱별로 지정할 수 있다.
- 창 메뉴와 우클릭 메뉴 구성을 사용자가 바꿀 수 있다.
- 앱별 배경, 패널 밀도, 표시 옵션을 저장할 수 있다.
- 패키지가 설치될 때 자신만의 아이콘, 창 설정, 권한, 런타임 UI를 함께 제공할 수 있다.
- 사용자가 만든 패키지와 위젯이 메인 화면의 일부가 될 수 있다.

## 3. 기본 디자인 유지 원칙

커스터마이징이 강해져도 기본 프리셋은 현재 디자인 방향을 유지한다.

기본 프리셋은 다음 요소를 기준으로 한다.

- 어두운 바탕
- 글래스모피즘 패널
- 반투명 표면
- blur 기반 깊이감
- 얇은 밝은 경계선
- compact window UI
- 파란 계열 기본 액센트
- 부드러운 hover/focus 효과
- 앱 아이콘 중심의 데스크톱
- 하단 작업표시줄

즉, 기본 상태는 지금의 프로토타입 디자인을 계승한다. 커스터마이징은 이 기본 상태를 파괴하는 기능이 아니라, 사용자가 자기 작업방식에 맞게 확장하는 기능이어야 한다.

## 4. 프리셋 시스템

UI 커스터마이징은 무작정 모든 값을 노출하면 복잡해진다. 따라서 기본은 프리셋 중심이어야 한다.

### 4.1 기본 프리셋

현재 디자인을 기준으로 한 기본 프리셋이다.

- 어두운 배경
- 블루 액센트
- 중간 blur
- 낮은 투명도
- compact taskbar
- glass window
- desktop icon grid

### 4.2 색상 프리셋

현재 Control Panel의 color preset 방향을 유지하되, 더 체계화한다.

예시는 다음과 같다.

- Standard Blue
- Cyber Neon
- Deep Forest
- Golden Hour
- Mono Glass
- Light Frost
- High Contrast

각 프리셋은 단순 색 하나가 아니라 다음 값을 묶어야 한다.

- wallpaper
- accent color
- glass opacity
- blur intensity
- border brightness
- taskbar style
- window titlebar style

### 4.3 사용자가 만든 프리셋

사용자는 현재 UI 상태를 저장해 자신의 프리셋으로 만들 수 있어야 한다.

저장 대상은 다음과 같다.

- 배경화면
- 배경 표시 방식
- accent color
- blur intensity
- transparency
- taskbar 위치와 밀도
- 창 기본 스타일
- 아이콘 크기
- 위젯 배치
- 시작 메뉴 구성

## 5. 바탕화면 커스터마이징

바탕화면은 단순히 앱 아이콘을 놓는 공간이 아니다.

My Web OS의 바탕화면은 사용자가 자신의 작업환경을 조립하는 첫 화면이다.

필요한 기능은 다음과 같다.

- 앱 아이콘 배치
- 파일/폴더 바로가기 배치
- 패키지 바로가기 배치
- 위젯 배치
- 아이콘 크기 조절
- 아이콘 정렬 방식 선택
- 그리드 간격 조절
- 여러 데스크톱 전환
- 데스크톱별 배경과 위젯 구성 저장
- 데스크톱별 작업 목적 지정

예를 들어 사용자는 다음과 같은 데스크톱을 만들 수 있다.

- 개발 데스크톱: Terminal, Code Editor, Docker, Logs
- 미디어 데스크톱: File Station, Media Player, Image Viewer, Model Viewer
- 운영 데스크톱: Resource Monitor, Package Center, Log Viewer, Service Manager
- 개인 데스크톱: 메모, 할 일, 사진, 음악 위젯

## 6. 시작 메뉴 커스터마이징

현재 Taskbar에는 시작 버튼 역할을 할 수 있는 아이콘이 있다. 이 버튼은 장기적으로 Start Menu의 진입점이 되어야 한다.

시작 메뉴는 단순 앱 목록이 아니라, 사용자의 Web OS를 탐색하는 중심 메뉴가 되어야 한다.

필요한 기능은 다음과 같다.

- 앱 목록
- 최근 사용 앱
- 즐겨찾기 앱
- 설치된 패키지
- 시스템 도구
- 파일 바로가기
- 전원/서버 조작
- 검색
- 사용자 지정 섹션
- 메뉴 레이아웃 프리셋

사용자는 시작 메뉴를 다음 방식으로 바꿀 수 있어야 한다.

- 섹션 추가/삭제
- 앱 순서 변경
- 고정 앱 설정
- 패키지 카테고리 표시
- 작은 메뉴/큰 메뉴 선택
- 텍스트 중심/아이콘 중심 선택
- 명령 팔레트형 모드 선택

## 7. 작업표시줄 커스터마이징

작업표시줄은 매일 쓰는 조작부이므로 커스터마이징이 매우 중요하다.

필요한 기능은 다음과 같다.

- 위치 선택: 하단, 상단, 좌측, 우측
- 높이/두께 조절
- 아이콘 크기 조절
- 검색 표시 방식 선택
- 알림 영역 표시/숨김
- 시간 표시 방식 선택
- 데스크톱 전환 표시 방식 선택
- 실행 중 앱 표시 방식 선택
- 고정 앱 영역 추가
- Agent 버튼 표시/숨김
- 투명도와 blur 별도 조절

기본값은 현재처럼 하단 compact taskbar를 유지한다.

## 8. 창 커스터마이징

Window는 My Web OS의 핵심 UX다. 앱은 모두 창 안에서 실행되므로 창의 커스터마이징은 단순 디자인 옵션이 아니라 작업성에 직접 영향을 준다.

필요한 기능은 다음과 같다.

- 앱별 기본 창 크기 저장
- 앱별 기본 위치 저장
- 앱별 최소 크기 설정
- 앱별 투명도 설정
- 앱별 배경 스타일 설정
- 창 타이틀바 스타일 변경
- 창 컨트롤 위치 선택
- 창 스냅 방식 설정
- 창 그림자 강도 설정
- 활성 창 강조 방식 선택
- 창별 always-on-top 설정

예를 들어 Terminal은 어두운 DOS풍 배경을 유지하고, Package Center는 정보 밀도가 높은 운영 콘솔형 창을 사용하며, Media Player는 거의 검은 배경의 viewer형 창을 사용하는 식이다.

## 9. 창 메뉴와 우클릭 메뉴

컨텍스트 메뉴는 사용자가 OS를 "직접 만지는 느낌"을 만드는 중요한 요소다.

필요한 기능은 다음과 같다.

- 바탕화면 우클릭 메뉴
- 파일 우클릭 메뉴
- 창 타이틀바 우클릭 메뉴
- 작업표시줄 우클릭 메뉴
- 앱 아이콘 우클릭 메뉴
- 패키지 우클릭 메뉴

메뉴는 고정된 것이 아니라 상황에 따라 달라져야 한다.

예시는 다음과 같다.

- 바탕화면: 새 패키지, 새 위젯, 배경 변경, 정렬, 새 바로가기
- 파일: 열기, 이름 변경, 공유, 바로가기 생성, 휴지통 이동
- 앱: 열기, 고정, 제거, 패키지 정보, 로그 보기
- 창: 최소화, 최대화, 투명도, 항상 위, 다른 데스크톱으로 이동
- 패키지: 실행, health check, backup, rollback, manifest 편집

## 10. 윈도우 백그라운드

윈도우 백그라운드는 앱 안쪽의 배경과 창 외곽의 표면을 모두 포함한다.

현재는 대부분 glass-effect와 어두운 배경을 사용한다. 장기적으로는 앱별 배경 타입을 설정할 수 있어야 한다.

필요한 옵션은 다음과 같다.

- Glass
- Solid
- Terminal Dark
- Media Black
- Document Light
- Dashboard Dense
- Transparent Overlay

앱별 기본값은 앱 성격에 맞게 다르게 잡는다.

- Terminal: Terminal Dark
- Media Player: Media Black
- Document Viewer: Document Light 또는 Dark Reader
- Package Center: Dashboard Dense
- Control Panel: Glass
- File Station: Glass + Dense
- Model Viewer: Media Black

## 11. 챗봇 캐릭터 방향

챗봇은 단순한 사이드 패널이 아니라 My Web OS의 캐릭터형 인터페이스가 되어야 한다.

현재 Agent 컴포넌트는 화면 우하단에 캐릭터 아바타와 말풍선을 표시하는 구조를 가진다. 이 구조를 확장하여 챗봇을 실제 작업 도우미로 만든다.

챗봇의 역할은 다음과 같다.

- 사용자와 대화한다.
- 시스템 상태를 알려준다.
- 파일/패키지/터미널 작업을 도와준다.
- 실패 상황을 설명한다.
- 명령 실행 전에 사용자에게 확인한다.
- LLM과 통신하는 대화 표면이 된다.
- CLI 출력과 LLM 응답을 말풍선/채팅창으로 번역한다.

캐릭터는 지나치게 방해하면 안 된다. 기본은 작게 존재하고, 필요할 때만 커진다.

## 12. 캐릭터 UI 구성

챗봇 캐릭터는 다음 UI 상태를 가져야 한다.

- idle: 대기
- listening: 사용자 입력 대기
- thinking: LLM 응답 생성 중
- executing: 명령 실행 중
- success: 작업 완료
- warning: 주의 필요
- error: 실패
- terminal: CLI 세션 연결 중

현재 Agent의 emotion 구조는 이 방향으로 확장할 수 있다.

캐릭터 표시 방식은 다음과 같다.

- 작은 원형 아바타
- 말풍선
- 확장 가능한 채팅 패널
- 명령 실행 상태 표시
- 감정/상태별 색상
- 알림과 구분되는 독립 레이어

장기적으로는 단순 아이콘이 아니라 Live2D, Inochi2D, Spine, CSS sprite, 또는 Canvas 기반 캐릭터로 확장할 수 있다.

## 13. 말풍선과 채팅창

챗봇 UI는 말풍선과 채팅창을 모두 가져야 한다.

말풍선은 짧은 상태 전달에 사용한다.

예시는 다음과 같다.

- 파일 업로드 완료
- 패키지 설치 성공
- Docker 연결 실패
- CPU 사용률 높음
- rollback 완료

채팅창은 긴 대화와 작업 기록에 사용한다.

채팅창 기능은 다음과 같다.

- 사용자 메시지
- 캐릭터 응답
- LLM 응답
- 명령 실행 요청
- 명령 출력 요약
- 실행 전 확인 버튼
- 작업 결과 카드
- 에러 설명
- 관련 앱 열기 버튼

말풍선은 순간적인 반응이고, 채팅창은 기록과 조작이 가능한 대화 공간이다.

## 14. CLI 래핑 방향

터미널은 지금 xterm 기반의 실제 CLI 앱으로 존재한다. 하지만 챗봇과 연결될 때는 CLI를 그대로 노출하는 것만으로는 부족하다.

필요한 방향은 다음과 같다.

- DOS풍 터미널 화면을 유지한다.
- CLI 입력/출력을 챗봇이 읽고 요약할 수 있다.
- 사용자가 자연어로 요청하면 CLI 명령 후보를 제안한다.
- 위험한 명령은 실행 전 확인한다.
- 명령 실행 결과를 말풍선과 채팅창으로 변환한다.
- 원하면 원본 터미널 로그도 볼 수 있다.

즉, CLI는 두 가지 모드로 보여야 한다.

- Raw Terminal Mode: 기존 터미널 그대로 사용
- Wrapped Assistant Mode: LLM/캐릭터가 CLI를 감싸고 해석하는 모드

## 15. DOS풍 인터페이스

사용자는 LLM과 통신할 때도 단순 웹 채팅창이 아니라, 터미널 특유의 분위기를 느낄 수 있어야 한다.

이를 위해 DOS풍 인터페이스를 제공한다.

디자인 방향은 다음과 같다.

- 어두운 배경
- 고정폭 폰트
- 프롬프트 라인
- 명령 히스토리
- 초록/청록/회색 계열 텍스트 옵션
- 커서 깜빡임
- scanline 또는 subtle glow 옵션
- 시스템 메시지와 사용자 메시지 구분

다만 너무 레트로해서 사용성을 해치면 안 된다. 기본은 현재 터미널의 어두운 스타일을 유지하고, DOS풍은 테마/모드로 제공한다.

## 16. LLM 통신 UI

LLM과의 통신은 단순 채팅이 아니라 OS 작업 흐름과 연결되어야 한다.

필요한 기능은 다음과 같다.

- 자연어 명령 입력
- 로컬 파일/패키지/터미널 맥락 전달
- 명령 실행 전 계획 표시
- 사용자 승인 후 실행
- 실행 결과 요약
- 실패 원인 설명
- 관련 앱 또는 파일 열기
- 다음 액션 제안

예시는 다음과 같다.

- "다운로드 폴더에서 큰 파일 찾아줘"
- "이 패키지 health check 해줘"
- "Docker 컨테이너 상태 정리해줘"
- "이 폴더를 zip으로 묶어줘"
- "최근 에러 로그 요약해줘"
- "이 파일을 이미지 뷰어로 열어줘"

챗봇은 사용자의 대신 로컬 PC를 조작할 수 있지만, 위험한 작업은 반드시 확인 흐름을 가져야 한다.

## 17. Agent와 Terminal의 통합 모델

Agent와 Terminal은 별개의 앱이지만, 경험적으로는 연결되어야 한다.

통합 모델은 다음과 같다.

- Terminal은 실제 명령 실행 표면이다.
- Agent는 명령을 설명하고, 제안하고, 결과를 요약하는 표면이다.
- LLM은 Agent의 두뇌 역할을 한다.
- Chat Panel은 대화와 실행 기록을 보존한다.
- Raw Output Panel은 원본 CLI 출력을 보존한다.

작업 흐름 예시는 다음과 같다.

1. 사용자가 Agent에게 "프로젝트 테스트 돌려줘"라고 말한다.
2. Agent가 실행할 명령 후보를 보여준다.
3. 사용자가 승인한다.
4. Terminal 세션 또는 백엔드 command runner가 명령을 실행한다.
5. Raw output은 터미널 로그에 남는다.
6. Agent는 결과를 요약해 말풍선과 채팅창에 표시한다.
7. 실패하면 관련 파일이나 로그 열기 버튼을 제공한다.

## 18. 챗봇 캐릭터의 위치와 UX

캐릭터는 항상 화면을 가리면 안 된다.

기본 위치는 우하단이 적절하다. 하지만 사용자가 위치를 바꿀 수 있어야 한다.

필요한 옵션은 다음과 같다.

- 표시/숨김
- 위치 변경
- 크기 변경
- 말풍선 자동 표시 여부
- 음성/효과음 여부
- 대화 패널 자동 열기 여부
- 터미널 연동 여부
- 시스템 알림 반응 여부

캐릭터는 "항상 떠 있는 귀여운 장식"이 아니라, 로컬 작업을 도와주는 조작 가능한 인터페이스여야 한다.

## 19. 사용자 흐름 예시

### 19.1 패키지 문제 해결

사용자가 패키지 센터에서 health check 실패를 본다.

흐름은 다음과 같다.

1. Agent가 실패 원인을 말풍선으로 요약한다.
2. 사용자가 "고칠 수 있어?"라고 묻는다.
3. Agent가 로그, manifest, runtime 상태를 확인한다.
4. 필요한 명령 또는 수정안을 제안한다.
5. 사용자가 승인하면 실행한다.
6. 실패하면 rollback 버튼을 제안한다.

### 19.2 파일 정리

사용자가 "다운로드 폴더 정리해줘"라고 말한다.

흐름은 다음과 같다.

1. Agent가 다운로드 폴더 스캔 계획을 보여준다.
2. 큰 파일, 오래된 파일, 중복 후보를 요약한다.
3. 삭제 대신 휴지통 이동을 제안한다.
4. 사용자가 승인하면 작업을 수행한다.
5. 결과를 말풍선과 채팅창에 보여준다.

### 19.3 Docker 점검

사용자가 "Docker 상태 봐줘"라고 말한다.

흐름은 다음과 같다.

1. Agent가 Docker Manager API를 호출한다.
2. 컨테이너 상태를 요약한다.
3. 멈춘 컨테이너나 오류 상태를 알려준다.
4. 재시작 후보를 제안한다.
5. 사용자가 승인하면 재시작한다.

## 24. 최종 방향

My Web OS의 프론트엔드는 단순히 OS처럼 보이는 화면이 아니다.

이 프로젝트의 프론트엔드는 사용자가 로컬 PC를 자기 방식으로 다시 구성하는 조작면이다.

따라서 UI/UX의 핵심은 다음과 같다.

- 현재의 글래스모피즘 디자인을 기본 정체성으로 유지한다.
- 커스터마이징은 배경색 변경 수준을 넘어 작업 흐름 변경까지 확장한다.
- 시작 메뉴, 작업표시줄, 바탕화면, 창, 창 배경, 우클릭 메뉴를 사용자가 다룰 수 있게 한다.
- 챗봇은 단순 보조 텍스트가 아니라 캐릭터 기반 OS 인터페이스가 된다.
- 터미널과 LLM 대화는 분리되지 않고, DOS풍 CLI와 말풍선/채팅창으로 함께 경험된다.
- 모든 변화는 "기존 OS보다 더 자유롭게 내 PC를 다루게 한다"는 목표에 맞아야 한다.

- Taskbar의 시작 버튼을 실제 Start Menu 진입점으로 만든다.
- Start Menu는 처음부터 과하게 만들지 말고 앱 목록, 최근 앱, 시스템 도구, Package Center, Settings, 전원/서버 조작 정도의 기본 구조로 시작한다.

- Control Panel에 UI 커스터마이징 섹션을 추가한다. 우선 Theme Presets, Desktop Layout, Taskbar, Window Style, Agent, Terminal Theme 순서가 적절하다.
- Window store에 앱별 기본값 모델을 추가한다. 최소 저장 대상은 `appId`, `defaultWidth`, `defaultHeight`, `defaultX`, `defaultY`, `opacity`, `backgroundMode`, `alwaysOnTop`이다.
- Agent store의 상태 이름을 기획서와 맞춘다. 현재 `idle/alert/error/happy/processing`을 `idle/listening/thinking/executing/success/warning/error/terminal`로 확장한다.

### 25.4 Agent / LLM / CLI 통합에서 필요한 변경

현재 Agent는 시각적 존재감은 있지만 실제 작업 도우미는 아니다. 다음 순서로 기능을 붙이는 것이 좋다.

1. Agent 채팅 패널을 만든다.
2. 메시지 저장 모델을 만든다. `agentStore`에 대화 목록, 실행 요청, 승인 대기 상태, 결과 카드를 추가한다.
3. 백엔드에 LLM 프록시 API를 추가한다. 예: `/api/ai/chat`
4. OS Action 매핑을 작게 시작한다. 예: 앱 열기, 파일 경로 열기, 패키지 health check, Docker 상태 조회, 최근 에러 로그 요약.
5. 위험 작업은 실행 전 승인 카드로 막는다. 삭제, overwrite 설치, rollback, 터미널 명령 실행은 반드시 승인 흐름을 거친다.
6. Terminal과 연결한다. Raw Terminal Mode는 유지하고, Wrapped Assistant Mode에서는 명령 후보, 실행 로그, 요약 결과를 Agent 채팅에 남긴다.

### 25.5 저장 구조 변경안

현재 `systemStore`, `windowStore`, `widgetStore`, `shortcutStore`가 일부 상태를 저장하지만, 커스터마이징 전체를 담기에는 부족하다.

추가할 상태 파일 후보는 다음과 같다.

- `theme-presets.json`: 기본/사용자 테마 프리셋
- `desktop-layouts.json`: 데스크톱별 배경, 아이콘, 바로가기, 위젯 배치
- `taskbar-settings.json`: 위치, 밀도, 아이콘, 검색, 트레이 표시
- `window-defaults.json`: 앱별 창 기본값과 배경 모드
- `start-menu-settings.json`: 섹션, 고정 앱, 레이아웃 모드
- `context-menu-settings.json`: 메뉴 항목 표시/숨김 및 순서
- `agent-settings.json`: 표시, 위치, 크기, 자동 말풍선, 터미널 연동
- `terminal-theme-settings.json`: raw/wrapped 모드와 DOS풍 테마
- `accessibility-settings.json`: 고대비, blur 감소, motion 감소, 캐릭터 숨김

### 25.6 구현 우선순위 재정리

P0는 "눈에 보이는 커스터마이징의 시작"과 "Agent의 실제 패널화"다.

- Start Menu 기본 구현
- Control Panel에 Theme Presets, Taskbar, Window Style, Agent, Terminal Theme 섹션 추가
- 데스크톱/작업표시줄/창 기본 설정 저장 모델 추가
- Agent 채팅 패널 기본 구현
- Agent 상태를 `idle/listening/thinking/executing/success/warning/error/terminal`로 확장

P1은 "작업 흐름 변경으로서의 커스터마이징"이다.

- 사용자 테마 프리셋 저장/불러오기
- 데스크톱 레이아웃 편집 모드
- 고정 앱/최근 앱 기반 Start Menu
- 앱별 window background 설정
- Terminal Wrapped Assistant Mode의 UI 골격
- CLI 실행 결과 요약 카드
- 위험 명령 승인 UI

P2는 "캐릭터형 OS 인터페이스"와 고급 개인화다.

- Agent 아바타 시스템 확장
- LLM 백엔드 프록시와 OS Action 매핑
- Package Center/Log Viewer/Terminal 결과를 Agent가 요약
- Context Menu 커스터마이징
- 데스크톱별 테마와 위젯 세트
- Live2D/Inochi2D/Canvas 캐릭터 검토
