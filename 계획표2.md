## 벤치마크 저격 전략 (Origin-OS / ArozOS / Puter) - 2026-04-21

### 1) 사용자 기대값 (이 프로젝트가 반드시 충족해야 하는 기준)
- 사용자는 "웹에서 돌지만 로컬 OS처럼 느껴지는 완성도"를 기대함.
- 사용자는 "예쁜 데모"보다 "끊김 없는 작업 흐름(실사용성)"을 기대함.
- 사용자는 "앱 실행"보다 "설치-업데이트-복구-롤백까지 되는 운영체제"를 기대함.
- 사용자는 기존 UI/브랜딩 유지 상태에서 기능 깊이가 증가하길 기대함.

### 2) 경쟁 프로젝트 핵심 강점 요약
- Origin-OS:
  - 강점: 웹 데스크톱 커스터마이즈 + 자체 스크립팅(OSL) 중심 생태계.
  - 위협 포인트: "사용자/개발자 확장성" 인식이 강함.
- ArozOS:
  - 강점: NAS/홈서버 실전 기능(스토리지, 네트워크, 파일서버, 보안 옵션) 폭이 넓음.
  - 위협 포인트: "운영 기능 폭"과 "시스템 관리자 관점" 완성도.
- Puter:
  - 강점: 인터넷 OS 포지셔닝 + 클라우드/앱스토어/자기호스팅 경험의 일관성.
  - 위협 포인트: "서비스형 사용자 경험"과 "생태계 접근성".

### 3) 본 프로젝트의 승리 조건 (UI 유지 전제)
- [ ] 현재 Glassmorphism/데스크톱 구조는 유지하고, UX는 흐름 기반으로만 강화.
- [ ] "앱센터"를 단순 설치기가 아니라 "운영 콘솔"로 승격.
- [ ] Host(로컬 자원) + Sandbox(격리 앱) 경계를 사용자에게 명확히 보이게 하되 조작은 단순하게 유지.
- [ ] 실패 상황(의존성 문제, 실행 실패, 업데이트 실패)에서 자동 복구/롤백을 기본 제공.

### 4) UI/UX 우선 원칙 (현재 UI 보존)
- [ ] 시각 리디자인 금지: 컴포넌트/레이아웃 틀 유지.
- [ ] 정보 밀도 개선: 상태 배지/이벤트 타임라인/복구 액션을 동일 컴포넌트 내 단계적 노출.
- [ ] 사용자 불안 제거: "왜 실패했는지" + "다음에 누를 버튼"을 항상 1스크린 내 제공.
- [ ] 긴 작업의 UX 표준화: 시작/중지/업데이트/롤백에 공통 진행상태, 타임아웃, 재시도 패턴 적용.

### 5) 기능 우위 로드맵 (경쟁 대비 초과 달성 목표)

#### P0. 운영 신뢰성 (즉시)
- [ ] 패키지별 Health Report를 앱센터 Installed 탭에 시각화.
- [ ] 이벤트/로그/복구 이력을 앱 단위 타임라인으로 통합.
- [ ] 복구 액션 원클릭: validate -> recover -> fallback rollback 자동 체인.

#### P1. 서비스 생애주기 완성
- [ ] 채널(stable/beta/alpha/canary)별 업데이트 정책 UI 추가.
- [ ] 업그레이드 전 자동 백업 기본 ON + 실패 자동 롤백 표준화.
- [ ] 의존성 충돌 시 설치 전 사전 차단 + 해결 가이드 노출.

#### P2. 생태계 우위
- [ ] 앱 제작자용 manifest 검증기/테스트 러너 제공.
- [ ] 스토어 품질 게이트(호환성, 리소스, 복구 가능성) 점수화.
- [ ] Host 브리지 권한 요청 UX를 표준화(사용자 승인/거절/기록).

### 6) 실행 판정 KPI (제품 관점)
- [ ] 앱 설치 성공률
- [ ] 업데이트 실패 후 자동 복구 성공률
- [ ] 런타임 장애 MTTR(평균 복구 시간)
- [ ] 사용자 액션당 완료 단계 수(클릭 수)
- [ ] 앱센터 내 "원인 불명 실패" 비율

### 7) 결론
- 이 프로젝트의 본질적 목표는 "웹 기반 데스크톱 UI"가 아니라, "웹에서 운영 가능한 컴퓨터 환경"이다.
- 따라서 경쟁 우위는 디자인 변경이 아니라, 운영 신뢰성/복구 자동화/생태계 품질에서 만든다.
- 현재 구현과 UI를 유지한 상태로도 충분히 추월 가능하며, 핵심은 앱센터를 운영 콘솔로 완성하는 것이다.

## 앱인앱 생태계 전략 (자체 Ecosystem 확장)

### 방향성
- 경쟁 프로젝트들이 강한 이유는 "OS 껍데기"가 아니라 "자체 앱 생태계"를 보유했기 때문.
- 우리도 동일하게, 앱센터를 단순 설치 UI가 아니라 **제작-배포-운영-복구**를 포함한 생태계 허브로 확장해야 함.
- 전제: 현재 UI/스타일은 유지하고, 기능 밀도와 운영 신뢰성으로 우위 확보.

### 핵심 전략
- [ ] **1st-party 도구군 확장**: 문서, 운영, 자동화, 개발 도구를 공식 패키지로 지속 제공.
- [ ] **앱 제작자 진입장벽 축소**: 템플릿 기반 생성, manifest 검증, 권한 가이드 자동화.
- [ ] **운영 품질 내재화**: 모든 앱에 health/log/event/rollback 표준을 강제.
- [ ] **품질 게이트 도입**: 스토어 등록 시 호환성/복구 가능성 기준을 통과해야 공개.
